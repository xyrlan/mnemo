## 1. Summary

v0.5.x addresses the pull retrieval gap in mnemo's MCP surface — but phases the
work so each step is justified by evidence, not assumption.

The original spec bundled three features into a single "pincer" PR: a TF-IDF-lite
search tool, a telemetry log, and a rich SessionStart payload. Critical review
revealed this inverts cause and effect: we don't have telemetry data proving the
pull path fails, yet we designed solutions assuming specific failure modes.

This revised spec splits the work into three sequential phases:

| Phase | Version | What | Why |
|-------|---------|------|-----|
| 1 | v0.5.2 | Project filter on legacy MCP tools | Bug fix: tools return vault-wide results despite per-project binding |
| 2 | v0.5.3 | MCP access log (telemetry) | Measurement: learn how Claude uses the pull path before optimizing it |
| 3 | v0.5.4 | Data-driven intervention (TBD) | Act on what the log reveals, not what we assume |

Phase 3 is deliberately undesigned — its scope depends on what Phases 1 and 2
reveal. This spec fully specifies Phases 1 and 2 only.

## 2. Problem and motivation

### 2.1 What's broken today (measured)

One thing is concretely broken and needs no telemetry to justify:

**The three MCP tools are vault-wide.** `list_rules_by_topic`, `read_mnemo_rule`,
and `get_mnemo_topics` scan all page types across `shared/` without filtering by
the current project. A rule synthesized from `bots/sg-imports/` sources surfaces
when Claude is working in the `mnemo` project. This directly contradicts the
per-project segregation invariant established in PR #22 and the binding user
memory: *"per-project segregation is correct, do NOT add global scope."*

The SessionStart hook already filters topics by project (via the rule-activation
index), making the inconsistency worse: the injection says "topics: [react, git]"
but a `list_rules_by_topic("database")` call returns rules from other projects.

### 2.2 What might be broken (suspected but unmeasured)

The original spec (§2.1) listed five weaknesses in the pull path:

1. Coarse 9-tag ontology forces imprecise retrieval
2. Two-step `list → read` cycle burns turn budget
3. Zero observability on what Claude queries and what fails
4. No recency or usage signal in ranking
5. Thin SessionStart injection (topic names only)

These are plausible hypotheses based on the authors' experience using mnemo, but
**we have zero telemetry confirming any of them**. The MCP counter
(`.mnemo/mcp-call-counter.json`) tracks daily aggregate counts — it cannot tell
us which topics Claude queries, which slugs it reads, or which queries return
zero hits.

Building a search engine (Feature A) and a rich injection payload (Feature C)
without this data is engineering by assumption. Phase 2 creates the measurement
infrastructure; Phase 3 acts on what it reveals.

### 2.3 What the push path already covers

PR #22's PreToolUse hook pushes rules at tool boundaries — zero model decision,
zero retrieval latency. After PR #22, the pull path's remaining use cases are
conversational and architectural queries ("what's our convention for X?"). These
are real but their frequency and failure modes are unknown.

## 3. Phase 1 — Project Filter (v0.5.2)

### 3.1 Goal

Fix the vault-wide leak in MCP tools. All three existing tools gain a `scope`
parameter defaulting to `"project"`. This is a **bug fix**, not a feature.

### 3.2 Design

#### 3.2.1 Project resolution

New private helper in `tools.py`:

```python
def _resolve_current_project(vault_root: Path) -> str | None:
    """Derive project name from cwd. Returns None on any failure."""
    try:
        from mnemo.core.agent import resolve_agent
        return resolve_agent(str(Path.cwd())).name
    except Exception:
        return None
```

Called once per request in `server.py:_handle_tool_call`, passed to all tool
functions.

#### 3.2.2 Project membership

New private helper in `tools.py`:

```python
def _rule_belongs_to_project(fm: dict, project: str) -> bool:
    """True if any source path starts with 'bots/<project>/'."""
    prefix = f"bots/{project}/"
    return any(s.startswith(prefix) for s in (fm.get("sources") or []))
```

Rules with no sources belong to no project — they only appear under
`scope="vault"`. This is correct: a rule without provenance has no project
binding.

#### 3.2.3 Tool changes

All three tools gain two new parameters with backwards-compatible defaults:

```python
def list_rules_by_topic(
    vault_root: Path, topic: str,
    *, scope: str = "project", project: str | None = None,
) -> list[RuleRef]:
```

When `scope == "project"` and `project` is not None, the inner loop adds a
filter: `if not _rule_belongs_to_project(fm, project): continue`.

When `scope == "project"` but `project` is None (resolution failed), effective
scope silently becomes `"vault"` — the pre-existing behavior. No regression.

`read_mnemo_rule` applies the same filter: a slug read under `scope="project"`
returns `None` if the rule doesn't belong to the current project. This prevents
cross-project reads even when Claude has a slug from a previous vault-wide query.

`get_mnemo_topics` under `scope="project"` collects tags only from rules that
pass `_rule_belongs_to_project`.

#### 3.2.4 Server changes

`_handle_tool_call` resolves `project` once, passes it to all dispatched tools:

```python
project = mcp_tools._resolve_current_project(vault_root)
scope = args.get("scope", "project")
# ... dispatch with scope=scope, project=project
```

`_TOOL_DEFS` updated: each tool's `inputSchema` gains an optional `scope` string
property (`"project"` | `"vault"`). Descriptions updated to mention project
scoping and `scope="vault"` opt-out.

#### 3.2.5 SessionStart — no change

The SessionStart hook already filters topics by project via the rule-activation
index. Phase 1 does not modify `session_start.py`. The injection text continues
to recommend `list_rules_by_topic(topic)` → `read_mnemo_rule(slug)`, which now
respects project scope.

### 3.3 Error handling

| Surface | Failure mode | Fallback |
|---------|-------------|----------|
| `_resolve_current_project` | `resolve_agent` raises | Return `None`; tools use vault-wide |
| `_rule_belongs_to_project` | Missing/malformed `sources` | Return `False`; rule excluded from project scope |
| Tool with `scope="project"`, project=None | Resolution failed | Silent vault-wide fallback |

### 3.4 Files touched

**Modified (2):**
- `src/mnemo/core/mcp/tools.py` — `scope`/`project` params, `_resolve_current_project`, `_rule_belongs_to_project`
- `src/mnemo/core/mcp/server.py` — per-call project resolution, `scope` in `_TOOL_DEFS` schemas + descriptions

**New (1):**
- `tests/unit/test_mcp_tools_project_filter.py`

### 3.5 Tests (~15)

**Non-negotiable invariant guards (4):**
1. `test_list_rules_by_topic_default_scope_is_project` — behavioral change pinned
2. `test_list_rules_by_topic_vault_scope_returns_all` — opt-out preserved
3. `test_read_mnemo_rule_project_scope_blocks_cross_project` — no cross-project reads
4. `test_get_mnemo_topics_project_scope_filters_tags` — tag union is project-scoped

**Additional:**
5. `test_resolve_current_project_returns_none_on_failure`
6. `test_rule_belongs_to_project_no_sources_returns_false`
7. `test_rule_belongs_to_project_mixed_sources`
8. `test_server_passes_project_to_tools`
9. `test_server_fallback_vault_when_project_none`
10. `test_tool_defs_include_scope_parameter`
11-15. Edge cases: empty vault, single-project vault, rule with sources from
multiple projects, topic with no project-matching rules, scope parameter
validation.

**Estimated delta:** ~15 new tests, ~755 total.

### 3.6 Backward compatibility

This is a **behavioral change**: callers of `list_rules_by_topic("react")` now
receive project-scoped results by default. Claude will see fewer rules per call.
This is intentional and correct — the previous behavior was a bug.

`scope="vault"` explicitly reproduces the old behavior. Tool descriptions
document this.

## 4. Phase 2 — MCP Access Log (v0.5.3)

### 4.1 Goal

Instrument every MCP tool call so we can answer, after ~2 weeks of dogfooding:

- Which topics does Claude query most? Which least?
- Which slugs get listed but never read? (Dead rules)
- Which queries return zero hits? (Ontology gaps)
- Does Claude use `scope="vault"` or stick with project default?
- How often does project resolution fail silently?

### 4.2 Design

#### 4.2.1 Log rotation extraction

`_rotate_if_needed` extracted from `rule_activation.py` to new
`core/log_utils.py` (~20 LOC). Pure refactor — `log_denial` and
`log_enrichment` import from the new location. Existing tests still pass
against the new location.

```python
# src/mnemo/core/log_utils.py
def rotate_if_needed(log_path: Path, max_bytes: int) -> None:
    """Rotate log_path → log_path.1 if exceeds max_bytes. Never raises."""
```

#### 4.2.2 Access log writer

New module `core/mcp/access_log.py` (~80 LOC):

```python
def record(vault_root: Path, entry: dict) -> None:
    """Append one JSON line to .mnemo/mcp-access-log.jsonl. Never raises."""
```

Behavior:
- Reads `injection.telemetry.enabled` from config; returns immediately if False
- Calls `rotate_if_needed` with configured max bytes
- Creates `.mnemo/` if missing
- Appends one JSON line, flushes
- Silent on all errors (disk full, permissions, serialization)

Entry schema:

```json
{
    "timestamp": "<ISO-8601-Z>",
    "tool": "list_rules_by_topic|read_mnemo_rule|get_mnemo_topics",
    "args": {"topic": "react"},
    "scope_requested": "project",
    "scope_effective": "project|vault",
    "project": "mnemo|null",
    "result_count": 3,
    "hit_slugs": ["rule-a", "rule-b", "rule-c"],
    "elapsed_ms": 12.4
}
```

Raw query content retained (per Q3 from original spec). Single string values
truncated at 1024 chars. Result fields derived from return value introspection.

#### 4.2.3 Server wiring

After each tool dispatch in `_handle_tool_call`, alongside the existing
`counter.increment`:

```python
access_log.record(vault_root, {
    "timestamp": datetime.now(timezone.utc).isoformat(),
    "tool": name,
    "args": _sanitize_args(args),
    "scope_requested": scope,
    "scope_effective": scope_effective,
    "project": project,
    "result_count": _count_results(result),
    "hit_slugs": _extract_slugs(result),
    "elapsed_ms": elapsed_ms,
})
```

`elapsed_ms` measured via `time.perf_counter()` around dispatch.

#### 4.2.4 Config flag

New sub-flag in DEFAULTS:

```python
"injection": {
    "enabled": False,
    "telemetry": {
        "enabled": True,
        "log": {"maxBytes": 1_048_576},
    },
},
```

Telemetry defaults to True (when injection is enabled). Users can disable
telemetry while keeping MCP retrieval features.

### 4.3 Error handling

| Surface | Failure mode | Fallback |
|---------|-------------|----------|
| `access_log.record` | Any exception | Silent pass; tool call unaffected |
| `rotate_if_needed` | OSError on rename | Silent pass; log grows until next attempt |
| `_sanitize_args` | Non-serializable value | Replace with `"<unserializable>"` |

## 5. Phase 3 — Data-Driven Intervention (v0.5.4)

### 5.1 Goal

After ~2 weeks of dogfooding v0.5.3, read `mcp-access-log.jsonl` and decide
what (if anything) to build next.

### 5.2 This phase is deliberately undesigned

We specify **decision criteria**, not solutions. The log data determines which
(if any) of the following interventions ship:

### 5.3 Decision framework

**Question A: Is the pull path used at all?**

Read the log. If total MCP calls over 2 weeks are < 20 (less than ~1.5/day),
the pull path is marginal. Intervention: improve SessionStart injection to
reduce pull dependency further, or accept that PR #22's push path handles the
load. Do NOT build `search_rules`.

**Question B: Are zero-hit queries common?**

Filter log entries where `result_count == 0`. If < 10% of queries, the 9-tag
ontology is adequate. If > 30%, the ontology has gaps. Intervention depends on
the nature of the gaps:
- Missing tags → add tags to existing rules (curation, not code)
- Queries that don't map to any tag → consider `search_rules` with simple
  substring match (NOT TF-IDF-lite; the vault has ~30 rules)
- Queries about topics with no rules → author new rules (curation, not code)

**Question C: Are there dead rules?**

Cross-reference `hit_slugs` across `list_rules_by_topic` and `read_mnemo_rule`
events. Rules that appear in list results ≥ 10 times but are never individually
read are candidates for description/tag rewriting. This is a curation signal,
not necessarily a code feature — a `mnemo doctor` check is justified only if the
pattern is systematic.

**Question D: Would a richer SessionStart help?**

Check how many tool calls happen in the first 3 entries of a session (proxy:
first minute of timestamps after a gap > 30 min). If Claude consistently calls
MCP tools early in sessions, it's compensating for thin injection. Intervention:
Top 3 rules with body previews in SessionStart. Ranking heuristic should use
telemetry data (read frequency) from day 1 — no placeholder `source_count`.

**Question E: Does `scope="vault"` get used?**

If Claude never passes `scope="vault"` explicitly, the opt-out was unnecessary
engineering (but harmless). If it does, understand why — it may indicate
cross-project rules that need better modeling.

### 5.4 Entry criteria for Phase 3

Phase 3 begins when ALL of:
- v0.5.3 has been live for ≥ 14 days
- `wc -l .mnemo/mcp-access-log.jsonl` ≥ 50 entries
- Author has read the raw log at least once

If the log has < 50 entries after 14 days, the pull path is not a priority.
Revisit in 30 days or after a vault growth event (new project onboarded).

### 5.5 Instrumentation: the `mnemo recall` suite (shipped 2026-04-17)

`src/mnemo/core/mcp/recall.py` + `mnemo recall` CLI implement the list→read
cross-reference that Questions B, C, and (partially) D rely on. Running
`mnemo recall` emits `.mnemo/recall-report.json` with `primacy_rate_at_{3,5,10}`,
MRR, p95 latency, a `misses` list (bootstrap→live-rank regressions), and a
`log_entries` field that surfaces distance to the §5.4 threshold.

Two practical consequences for Phase 3 planning:

- Questions B and C can be answered directly from the report instead of
  writing ad-hoc scripts. `misses` maps to Question C (slugs that were
  consumed historically but no longer rank near the top); tabulating
  `result_count == 0` entries from `.mnemo/mcp-access-log.jsonl` still needs
  a separate pass for Question B.
- The metric is called *primacy*, not *hit rate*, because
  `list_rules_by_topic` does not truncate. The recall report measures how
  often the consumed slug ended up in the first N positions, i.e. primacy
  bias against the full response — it does *not* measure whether Claude
  saw the rule.

Do NOT treat a low `primacy_rate_at_5` as justification for jumping the §5.4
gate. The gate exists precisely because small corpora produce noisy primacy
numbers.

## 6. Scope and non-goals

### 6.1 Invariants preserved (all phases)

- `dependencies = []` — all code is pure stdlib
- Hand-rolled parsing — no new YAML/TOML/search library
- Fail-open absolute — no new code path propagates exceptions to Claude Code
- Per-project segregation — all user-visible retrieval is project-scoped by default
- Auto-populating over manual curation — no new user commands for day-to-day operation

### 6.2 Explicit non-goals

- **Semantic search via embeddings** — violates `dependencies = []`; not justified
  for a ~30-rule corpus
- **TF-IDF-lite scoring engine** — over-engineered for corpus size; if substring
  match is needed (Phase 3 Question B), implement as simple case-insensitive
  substring match over `name + description + body`, ~15 LOC
- **LLM-powered result re-ranking** — latency/tokens not justified
- **Deep tag hierarchy** — violates controlled-vocabulary binding
- **Per-project context via MCP protocol** (`initialize` params) — `Path.cwd()`
  per-call is sufficient and more robust
- **`mnemo doctor` checks** — deferred to Phase 3; only justified if log data
  shows systematic patterns

### 6.3 Decisions inherited from original spec (not re-litigated)

| # | Decision | Kept? | Note |
|---|----------|-------|------|
| Q1 | Project filter on all MCP tools | Yes | Phase 1 core |
| Q2 | Top 3 scope = enrich-only | Deferred | Phase 3 Question D |
| Q3 | Raw queries in telemetry | Yes | Phase 2 core |
| Q4 | Top 3 + topic list coexist | Deferred | Phase 3 Question D |
| Q5 | Counter + JSONL coexist | Yes | Phase 2 core |
