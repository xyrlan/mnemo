# mnemo v0.2 — LLM Extraction Design

**Version:** v0.2 design
**Date:** 2026-04-13
**Status:** Draft (awaiting implementation plan)
**Author:** xyrlan
**Predecessor:** [v0.1 design](2026-04-11-mnemo-design.md) (shipped to master, dogfood-validated)

---

## 1. Summary

v0.2 adds **LLM-powered extraction of canonical pages** from Tier 1 mirrored memory into `shared/_inbox/`, materializing the "auto-populating second brain" promise that v0.1 deliberately punted.

**Scope boundary:** v0.2 is *only* LLM extraction. Multi-machine sync (Q4), backup strategy (Q5), and plugin auto-update (Q6) from the v0.1 open questions are explicitly out of scope — they are orthogonal to extraction and can evolve independently later.

**Input source:** `bots/<agent>/memory/*.md` only. Daily logs (`bots/<agent>/logs/*.md`) are explicitly deferred to v0.3 — their signal-to-noise is too low to justify the token spend, and memory files are already in the target canonical format.

**Trigger model:** manual `/mnemo extract` command + passive hint emitted during `SessionEnd` when ≥ 5 new memory files accumulate since the last extraction. No background execution, no daemons — the hooks-only invariant of v0.1 is preserved.

**Execution:** `claude --print` subprocess reusing the user's existing Claude Code authentication (Pro/Max subscription OR API key). Zero new runtime dependencies — the v0.1 stdlib-only invariant is preserved. Power users can opt into direct Anthropic API via `extraction.preferAPI: true`.

**Output contract:** extracted pages land in `shared/_inbox/<type>/<slug>.md` where `<type>` is one of `feedback`, `user`, `reference`, `project`. The plugin **never touches** `shared/<type>/<slug>.md` (user-curated territory) — the inbox staging area is the entire plugin-writable surface inside `shared/`.

**Re-extraction is content-addressable:** a state file at `~/mnemo/.mnemo/extraction-state.json` tracks source hashes, written hashes, and per-slug status (`inbox` / `promoted` / `dismissed`). The core invariant is **re-extraction never destroys review work in progress** — if the user has hand-edited a file in `_inbox/`, the next run writes a `.proposed.md` sibling instead of overwriting.

**Tagline reaffirmed:** "The Obsidian that populates itself so your Claude never forgets" — v0.1 captured the raw material; v0.2 is where "populates itself" stops being aspirational.

## 2. Problem and motivation

v0.1 shipped the capture pipeline. After one week of personal dogfood, the author's vault contains:

- 13 memory files across 3 active agents (`central-inteligencia-frontend`, `sg-imports`, `clubinho`)
- Several daily logs with real session traces
- **An empty `shared/people.md`** — the only file in `shared/`

The empty `shared/` is not a bug in v0.1. It's a deliberate consequence: v0.1 was pure capture, Tier 2 curation was left manual. But the empty `shared/` is also the whole pain point of the second-brain category: **manual curation doesn't happen** even when the raw material is already there and the user cares. v0.2 exists because that gap is the difference between a log aggregator and a PKM tool.

### 2.1 — The insight that shapes v0.2

The Claude Code memory files mirrored by v0.1 into `bots/<agent>/memory/*.md` **are already in the target canonical format** for `shared/`. They have:

- YAML frontmatter with `name`, `description`, `type`
- Body with `**Why:**` and `**How to apply:**` sections
- Types matching PKM categories: `user` / `feedback` / `project` / `reference`

This observation collapses what looked like "LLM extraction from noisy logs" (hard, expensive, low ROI) into "cross-agent clustering and promotion of already-structured content" (tractable, cheap, high ROI). The LLM's job is narrow: group related memories across agents, produce one canonical page per conceptual entity, preserve the source format.

### 2.2 — Why memory-only (not logs, not transcripts)

Four input sources were considered:

| Source | Decision | Reason |
|---|---|---|
| Mirrored memory files | **v0.2 ✓** | Already in target format; LLM job is clustering/merging, not extraction |
| Daily logs (timeline) | Deferred to v0.3 | ~80% noise (`teste`, `c`, `sim`, file-edit churn); low signal per token |
| Session transcripts | Deferred indefinitely | Not captured in v0.1; requires new plumbing; violates "small capture" ethos |
| Hybrid (memory + logs) | Rejected | Doubles complexity to chase the 20% of valuable content in logs |

Shipping memory-only first also validates the extraction architecture against a clean input before investing in noise reduction. If memory extraction produces high-quality `_inbox/` pages in real use, v0.3 can layer logs on top. If it doesn't, we learn that before paying the v0.3 cost.

## 3. Goals and non-goals

### 3.1 — Goals (v0.2)

1. **Auto-populate `shared/`** from Tier 1 memory files with zero ongoing user action beyond running `/mnemo extract` (or accepting the passive hint).
2. **Preserve v0.1 invariants.** Hooks-only, stdlib-only, cross-platform, local-first, private, non-invasive, robust-by-default, Obsidian-optimal.
3. **Subscription-first auth.** The common path reuses the user's existing Claude Code authentication — no new credentials required.
4. **Never destroy review work.** Re-extraction is safe against hand-edited `_inbox/` files, dismissed slugs, and already-promoted pages.
5. **Cross-agent clustering.** Memories about the same conceptual entity across different agents merge into a single canonical page.
6. **Cost-transparent.** Users see token usage and (when applicable) dollar cost of every run. No hidden spend.
7. **Partial success tolerance.** A run with 1 failed chunk out of 5 returns 4 successful chunks on disk plus a retryable next run, not a rollback.

### 3.2 — Non-goals (v0.2)

1. **No log-based extraction.** Daily logs are not a v0.2 input. Deferred to v0.3.
2. **No background execution.** No cron, no daemons, no filesystem watchers. Extraction runs when the user asks (or is gently hinted to ask).
3. **No automatic promotion of cluster types.** For `feedback`, `user`, and `reference`, pages land in `_inbox/<type>/` with `#needs-review`. The user decides when to move them to `shared/<type>/<slug>.md`. The plugin-writable surface inside `shared/` is precisely `shared/_inbox/**` plus the deliberate carve-out `shared/project/**` (see §4.4 and §6.5). The plugin never writes to `shared/feedback/**`, `shared/user/**`, or `shared/reference/**`.
4. **No live diff review.** Proposed updates to already-promoted pages are written as `.update-proposed.md` files — not as interactive 3-way merges.
5. **No multi-machine sync.** Out of scope. User can use git/Syncthing/etc on the vault.
6. **No backup automation.** Out of scope. User data is sacred and user-managed.
7. **No plugin auto-update.** Out of scope.
8. **No SDK dependency.** The Claude Agent SDK is explicitly not adopted in v0.2. Rationale in §6.
9. **No semantic quality validation.** LLM hallucinations and bad clusters are caught via human review of `_inbox/`. Testing covers structural correctness only.

## 4. Architecture

### 4.1 — High-level pipeline (single `/mnemo extract` run)

```
/mnemo extract
   │
   ▼
┌─────────────────────────────────────────────────────────┐
│ scanner: walk bots/*/memory/*.md                        │
│  → groups by type {feedback, user, reference, project}  │
│  → diff vs extraction-state.json (skip unchanged)       │
└────────────┬────────────────────────────────────────────┘
             │
             ├──▶ project files ──▶ promote.py (1:1, no LLM)
             │                           │
             │                           ▼
             │              shared/project/<agent>__<slug>.md
             │
             ├──▶ feedback files ─┐
             ├──▶ user files ─────┤──▶ prompts.py builds 1 prompt per type
             └──▶ reference files ┘   (chunked: ≤10 files per call)
                                                │
                                                ▼
                                    core/llm.py (subprocess)
                              claude --print --model haiku-4-5
                                                │
                                                ▼
                                  JSON {pages: [{slug, ...}]}
                                                │
                                                ▼
                                         inbox.py writes
                                                │
                                                ▼
                              shared/_inbox/<type>/<slug>.md
                                                │
                                                ▼
                            update extraction-state.json (atomic)
                                                │
                                                ▼
                              print summary (calls, time, cost)
```

### 4.2 — Hint pipeline (passive, in `SessionEnd` hook)

```
SessionEnd hook fires
   │
   ▼
existing v0.1 steps (mirror, log line, cache clear)
   │
   ▼
_maybe_emit_hint(cfg):
    load extraction-state.json (silent no-op on failure)
    count files in bots/*/memory/*.md with mtime > state.last_run
    if count >= cfg.extraction.hintThreshold (default 5):
        if today's log does not already contain a hint line:
            append "🟡 N new memories — run /mnemo extract"
        (if count >= threshold * 3: enphatic variant)
   │
   ▼
return 0 (hook never fails the session)
```

**Hint is cosmetic.** It never runs extraction, never calls the LLM, never mutates state. It only reads state and appends a log line. Any exception is caught by the v0.1 error infrastructure and silenced.

### 4.3 — Invariants preserved from v0.1

- **Hooks-only.** Extract is a manual command; the only new hook code is a read-only append inside `SessionEnd`.
- **Stdlib-only.** `subprocess`, `json`, `pathlib`, `hashlib`, `re` — all stdlib.
- **Cross-platform.** Subprocess, atomic file operations, same code on Linux/macOS/WSL. Best-effort native Windows.
- **Local-first.** The LLM subprocess is the only outbound traffic, initiated by user action, visible in config.
- **Non-invasive.** Plugin touches `shared/_inbox/`, `shared/project/` (1:1 promoted files), `~/mnemo/.mnemo/extraction-state.json`, and appends to existing daily logs. Nothing else.
- **Robust by default.** Hooks still exit 0 unconditionally. Extraction (as a manual command) fails loudly but cleanly.
- **Circuit breaker interacts correctly.** Extract errors are logged to `.errors.log` with prefix `extract.*` and are filtered out of the hook circuit breaker count (extraction failures must never disable capture hooks).

### 4.4 — Invariants introduced by v0.2

- **`shared/_inbox/` is plugin-writable territory.** The plugin may create, overwrite-if-unchanged, sibling-write-if-edited, and read files inside it. The user may edit freely; the plugin respects those edits via content-addressing.
- **`~/mnemo/.mnemo/` is plugin-writable territory.** A dot-directory the user is not expected to touch. Contains `extraction-state.json` (and future v0.2.x state files if needed).
- **`shared/project/` holds 1:1 promoted memories.** Namespaced by agent (`<agent>__<slug>.md`). No clustering. **This is a deliberate plugin-writable carve-out** — the only `shared/<type>/` subdir the plugin writes to outside `_inbox/`. Content-addressable (source hash tracked, user edits detected via written_hash, conflicts produce `.proposed.md` siblings).
- **`shared/feedback/`, `shared/user/`, `shared/reference/` are sacred.** User-promoted canonical pages for cluster types. The plugin reads them to detect "already promoted" status but **never writes** to them. If a source memory that corresponds to an already-promoted slug changes, the plugin writes `_inbox/<type>/<slug>.update-proposed.md` — never touches the canonical file.
- **Re-extraction preserves review work.** See the decision table in §5.4.

### 4.5 — Three-tier model (unchanged from v0.1 semantically)

Tier 1 remains the inbox (raw capture, plugin-managed). Tier 2 remains the truth layer (canonical facts). Tier 3 remains published knowledge. v0.2 only changes **how** Tier 2 gets populated: instead of manual curation only, it now has a plugin-driven staging area (`shared/_inbox/`) that the user promotes from.

**`shared/` layout change from v0.1:** the v0.1 spec speculatively defined four subdirs (`people/`, `companies/`, `projects/`, `decisions/`) based on a second-brain taxonomy. After dogfood, only `shared/people.md` exists (empty). The v0.2 layout replaces the speculative taxonomy with one that mirrors the actual memory types:

```
~/mnemo/shared/
├── _inbox/                      ← plugin-writable staging
│   ├── feedback/
│   ├── user/
│   └── reference/
├── feedback/                    ← user-promoted canonical
├── user/
├── reference/
├── project/                     ← 1:1 promoted (plugin-writable)
│   └── <agent>__<slug>.md
└── people.md                    ← legacy from v0.1, ignored
```

`shared/people.md` is left in place (per the "user data is sacred" rule from v0.1 decision #15) and documented as legacy in `/mnemo doctor`. The empty file is harmless.

## 5. Data model and decision tables

### 5.1 — Memory file recognition

The scanner walks `bots/*/memory/*.md` and, for each file:

1. Skips `MEMORY.md` (it's an auto-generated index file).
2. Parses YAML frontmatter.
3. Reads `type` field; if absent or unrecognized, defaults to `feedback` and emits a warning into `.errors.log` (with `where: "extract.scan"`).
4. Computes `source_hash = sha256(file_content_bytes)`.
5. Classifies into one of the four buckets: `feedback`, `user`, `reference`, `project`.

### 5.2 — Slug generation

Slug is derived from the memory file's stem (filename without extension), normalized:

- lowercased
- non-`[a-z0-9]` runs replaced with `-`
- leading/trailing `-` stripped
- truncated to 60 characters (at word boundary if possible)

Example: `feedback_no_commit_without_permission.md` → slug `no-commit-without-permission`.

Type prefix is stored in the path (`_inbox/feedback/no-commit-without-permission.md`), not the slug.

For cluster outputs (LLM-generated pages), the slug comes from the LLM response's `slug` field, normalized through the same rules. The LLM is instructed in the prompt to generate stable, descriptive slugs.

### 5.3 — State file schema

Location: `~/mnemo/.mnemo/extraction-state.json`

```json
{
  "schema_version": 1,
  "last_run": "2026-04-13T15:30:00",
  "entries": {
    "feedback/no-commit-without-permission": {
      "source_files": [
        "bots/central-inteligencia-frontend/memory/feedback_no_commits.md",
        "bots/clubinho/memory/feedback_no_commit_without_permission.md"
      ],
      "source_hash": "sha256:abc...",
      "written_hash": "sha256:def...",
      "written_at": "2026-04-13T15:30:00",
      "status": "inbox"
    }
  }
}
```

**Entry status values** (stored):
- `inbox` — plugin wrote the page to `shared/_inbox/<type>/<slug>.md` for LLM-generated cluster types (`feedback`, `user`, `reference`). File currently exists in `_inbox/`.
- `promoted` — the slug now exists at `shared/<type>/<slug>.md` (user moved it from `_inbox/`). Plugin will no longer overwrite `_inbox/` for this slug; new source changes produce `.update-proposed.md` siblings. Applies only to cluster types.
- `dismissed` — the slug was previously in `_inbox/`, but the file is no longer there and no corresponding `shared/<type>/<slug>.md` exists. Interpretation: the user deleted it. Plugin will not recreate unless `--force` is passed. Applies only to cluster types.
- `direct` — project type only. Plugin wrote the page directly to `shared/project/<agent>__<slug>.md` (no `_inbox/` staging). File currently exists at that path.

**Derived (runtime-only) state**:
- `inbox_edited` — stored status is `inbox` but on-disk hash != `written_hash`. The user hand-edited the inbox file. Any source change produces a `.proposed.md` sibling.
- `direct_edited` — stored status is `direct` but on-disk hash != `written_hash`. The user hand-edited the promoted project file directly. Any source change produces a `.proposed.md` sibling inside `shared/project/`.

**Schema versioning:** `schema_version` is read on load. Unknown versions cause an abort with a clear message. v0.2 ships at `1`.

### 5.4 — Re-extraction decision table

For each cluster the pipeline produces (either LLM-generated or project 1:1):

| `source_hash` changed? | entry in state? | file in `_inbox/`? | on-disk hash == `written_hash`? | file in `shared/<type>/`? | Classification | Action |
|---|---|---|---|---|---|---|
| no | — | — | — | — | unchanged | skip silently |
| yes | no | no | — | — | fresh | write `_inbox/<type>/<slug>.md`, record entry with status=`inbox` |
| yes | yes (`inbox`) | yes | yes | — | overwrite_safe | overwrite, update `written_hash`, record `written_at` |
| yes | yes (`inbox`) | yes | **no** | — | sibling_proposed | write `_inbox/<type>/<slug>.proposed.md`, leave original, do NOT update `written_hash` of original, record conflict in run summary |
| yes | yes (`inbox`) | no | — | yes | promoted_updated | transition entry to `promoted`, write `_inbox/<type>/<slug>.update-proposed.md` |
| yes | yes (`inbox`) | no | — | no | dismissed | transition entry to `dismissed`, skip |
| yes | yes (`promoted`) | — | — | yes | promoted_updated | write `_inbox/<type>/<slug>.update-proposed.md` (preserves `promoted` status) |
| yes | yes (`promoted`) | — | — | no | legacy promoted | user deleted the promoted file; treat as `dismissed`, log warning |
| yes | yes (`dismissed`) | — | — | — | dismissed_skip | skip |

**`--force` flag:** treats all `dismissed` and `promoted` entries as if they were fresh. Writes to `_inbox/` unconditionally (still non-destructive to user edits — the `sibling_proposed` rule still applies when on-disk hash mismatches).

### 5.4.1 — Decision table for `project` type entries (`status=direct`)

Projects bypass `_inbox/` and write directly to `shared/project/<agent>__<slug>.md`. Their decision table is simpler because there is no staging-vs-promoted distinction:

| `source_hash` changed? | entry in state? | file at `shared/project/<agent>__<slug>.md`? | on-disk hash == `written_hash`? | Classification | Action |
|---|---|---|---|---|---|
| no | — | — | — | unchanged | skip silently |
| yes | no | — | — | fresh | write file, record entry with `status=direct` |
| yes | yes (`direct`) | yes | yes | overwrite_safe | overwrite, update `written_hash` and `written_at` |
| yes | yes (`direct`) | yes | **no** | sibling_proposed | write `shared/project/<agent>__<slug>.proposed.md`, leave original, do NOT update `written_hash`, record conflict |
| yes | yes (`direct`) | no | — | dismissed | user deleted the file; transition entry to internal `project_dismissed` marker, skip |

**`--force` for project entries:** treats `project_dismissed` entries as fresh and re-writes. `sibling_proposed` rule still applies.

### 5.5 — Post-process dedupe (chunks produce same slug)

When a type has more than `chunkSize` files, it is split into chunks, each going through its own `llm.call()`. Different chunks may produce the same slug for clusters that conceptually span both (e.g., 11 feedback files → chunks of [10, 1], and the 11th file is about the same concept as one in the first chunk).

After all chunks of a type complete, the orchestrator deduplicates by slug:

1. Group all emitted pages by slug.
2. For each group with >1 page:
   - Merge `source_files` as the union.
   - Use the `body` from the page with the most source files (heuristic — larger cluster likely has better grounding).
   - If tied, use the earlier chunk's body.
   - Record the merge in the run summary.
3. Pass the deduped list to `inbox.apply_pages()`.

## 6. Components

The v0.2 code lives in a new sub-package `src/mnemo/core/extract/` plus one sibling module `src/mnemo/core/llm.py`. Modifications to existing files are additive only.

### 6.1 — New: `core/llm.py` (~95 LOC)

Thin wrapper over `claude --print` subprocess. **This is the only new module outside the `extract/` sub-package — it is intentionally reusable for v0.3 (log-based extraction) and beyond.**

**Public interface:**
```python
@dataclass(frozen=True)
class LLMResponse:
    text: str                     # the extracted/cleaned content
    total_cost_usd: float | None  # from the result envelope, None if absent
    input_tokens: int | None
    output_tokens: int | None
    api_key_source: str | None    # "none" for subscription, "user"/"env" otherwise
    raw: dict                     # full parsed envelope, for debugging

class LLMSubprocessError(Exception): ...
class LLMParseError(Exception): ...

def call(
    prompt: str,
    *,
    system: str | None,
    model: str = "claude-haiku-4-5",
    timeout: int = 60,
) -> LLMResponse: ...
```

**Subprocess construction:**
```
claude --print --no-session-persistence --output-format json
       --model <model> --tools ""
       [--system-prompt <system>]
```

Prompt is piped via stdin. Stderr captured separately. Intentionally **does not use `--bare`** (decision #16 from v0.1 — `--bare` is incompatible with subscription auth).

**Resilient JSON parsing** (the core of the module):

```python
def _parse_llm_json(result_text: str) -> dict:
    s = result_text.strip()
    if s.startswith("```"):
        s = re.sub(r"^```(?:json)?\s*", "", s)
        s = re.sub(r"\s*```\s*$", "", s)
    start = s.find("{")
    if start < 0:
        raise LLMParseError("no JSON object found in response")
    try:
        obj, _end = json.JSONDecoder().raw_decode(s[start:])
        return obj
    except json.JSONDecodeError as e:
        raise LLMParseError(
            f"malformed JSON at offset {e.pos}: {e.msg}"
        ) from e
```

This handles: fenced output, unfenced output, output with prose prefixes ("Here is the JSON:"), output with trailing commentary, and nested braces in strings (via `raw_decode`).

**Failure handling:**

| Condition | Action |
|---|---|
| `FileNotFoundError` on `claude` CLI | Raise `LLMSubprocessError("claude CLI not found; install Claude Code first")` — orchestrator aborts whole run |
| `subprocess.TimeoutExpired` | Kill process; 1 retry with 2s backoff; on second failure raise `LLMSubprocessError` |
| `returncode != 0` with stderr containing `rate.?limit` (case-insensitive) | 1 retry with 5s backoff; on second failure raise `LLMSubprocessError` with stderr |
| `returncode != 0` other | Raise `LLMSubprocessError` with stderr (no retry) |
| `json.JSONDecodeError` on outer envelope | Raise `LLMParseError` (no retry) |
| `_parse_llm_json` raises | Propagate `LLMParseError` (no retry — prompt bug, not transient) |

All other exceptions (including `KeyboardInterrupt`) propagate. **No `except Exception` blanket in `core/llm.py`.**

**Captured fields from the envelope:**
- `result` → `text` (after fence/prose stripping)
- `total_cost_usd` → propagated as-is (authoritative; never hardcoded)
- `usage.input_tokens`, `usage.output_tokens` → propagated
- `apiKeySource` → propagated (for cost UX decisions)

### 6.2 — New: `core/extract/scanner.py` (~80 LOC)

**Public interface:**
```python
@dataclass(frozen=True)
class MemoryFile:
    path: Path
    agent: str
    type: str   # one of feedback|user|reference|project
    frontmatter: dict
    body: str
    source_hash: str

@dataclass(frozen=True)
class ScanResult:
    by_type: dict[str, list[MemoryFile]]
    unchanged_slugs: set[str]   # slugs whose source_hash matches state
    dirty_files: list[MemoryFile]

def scan(vault_root: Path, state: ExtractionState) -> ScanResult: ...
```

Walks `bots/*/memory/*.md`, skipping `MEMORY.md`. Parses frontmatter (reusing the `_FRONTMATTER_RE` pattern from `core/wiki.py` for consistency). Classifies by `type`, defaulting unknown/missing to `feedback` with a warning logged to `.errors.log`.

`source_hash` is SHA256 of raw file bytes. Used both for change detection and (aggregated per slug) for state tracking.

Pure-ish: reads files, does not write. Does not invoke the LLM.

### 6.3 — New: `core/extract/prompts.py` (~140 LOC)

Three specialized prompt builders plus a chunking helper:

```python
def build_feedback_prompt(files: list[MemoryFile]) -> str: ...
def build_user_prompt(files: list[MemoryFile]) -> str: ...
def build_reference_prompt(files: list[MemoryFile]) -> str: ...

def chunks_for(
    files: list[MemoryFile],
    chunk_size: int,
) -> Iterator[list[MemoryFile]]: ...

FEEDBACK_SYSTEM_PROMPT: str = "..."
USER_SYSTEM_PROMPT: str = "..."
REFERENCE_SYSTEM_PROMPT: str = "..."
```

Each prompt contains:
1. Task instructions (cluster-by-entity, produce canonical pages, preserve source format).
2. Output schema as inline JSON example: `{"pages": [{"slug", "name", "description", "type", "body", "source_files"}]}`.
3. 2 few-shot examples (short, hand-crafted, cover the "merge cross-agent" and "single-source pass-through" cases).
4. File separator format: `<<<FILE: <path>>>>\n<frontmatter+body>\n<<<END>>>` — lets the LLM reference exact source paths.
5. Concatenated input files.

Prompts are **Python string constants** (not external template files). Rationale: consistency with the v0.1 `templates/` directory which only holds static markdown for user-visible scaffolding, never runtime-loaded logic. String constants are greppable, version-controllable inline, and don't introduce a template loading layer.

**Chunking:** `chunks_for` splits `files` into sub-lists of length ≤ `chunk_size` (default 10), preserving scanner discovery order. **Chunking does not re-sort** — cross-chunk slug collisions are handled by the post-process dedupe in §5.5, not by pre-sorting.

### 6.4 — New: `core/extract/inbox.py` (~140 LOC)

The densest module — implements the decision table (§5.4) and the write pipeline.

```python
@dataclass
class ExtractedPage:
    slug: str
    type: str
    name: str
    description: str
    body: str
    source_files: list[str]
    source_hash: str   # computed after dedup

@dataclass
class ApplyResult:
    written_fresh: list[str]
    overwrite_safe: list[str]
    sibling_proposed: list[tuple[str, str]]   # (slug, sibling_path)
    update_proposed: list[str]
    dismissed_skipped: list[str]
    unchanged_skipped: list[str]
    errors: list[tuple[str, Exception]]

def apply_pages(
    pages: list[ExtractedPage],
    state: ExtractionState,
    vault_root: Path,
) -> ApplyResult: ...

def dedupe_by_slug(pages: list[ExtractedPage]) -> list[ExtractedPage]: ...

def atomic_write_state(
    state: ExtractionState,
    path: Path,
) -> None: ...
```

`apply_pages` classifies each page via the §5.4 decision table, executes the action, and updates the state in-memory. **State persistence is done by the orchestrator after each chunk**, not inside `apply_pages` — this keeps `inbox.py` testable without filesystem assumptions about state location.

`atomic_write_state` writes to `<path>.tmp`, then `os.replace` to `path`. Atomic on POSIX and Windows NTFS.

**File writes in `_inbox/`** use the same atomic pattern:
- Write body + frontmatter to `<target>.tmp`
- `os.replace(tmp, target)`
- If target already existed, the rename replaces it atomically

**Frontmatter emitted for output pages:**
```yaml
---

description: <from cluster>
type: <feedback|user|reference|project>
extracted_at: 2026-04-13T15:30:00

```

The `#needs-review` tag is applied via frontmatter `tags` array — Obsidian indexes it for the user's review workflow.

**No `except Exception` blanket.** Typed exceptions only: `ExtractionIOError` for filesystem failures, propagated to the orchestrator. `KeyboardInterrupt` propagates freely.

### 6.5 — New: `core/extract/promote.py` (~50 LOC)

Handler for the `project` type. Zero LLM calls, zero clustering.

```python
def promote_projects(
    files: list[MemoryFile],
    state: ExtractionState,
    vault_root: Path,
) -> ApplyResult: ...
```

For each file:
1. Compute target: `shared/project/<agent>__<slug>.md` (direct, no `_inbox/` staging).
2. Compute `source_hash` from single-file content.
3. Look up state; apply the **project decision table** (§5.4.1, below).
4. Write via the atomic pattern.
5. Record entry in state with `status=direct`.

**Rationale for bypassing `_inbox/`:** project memories are already agent-scoped (the filename namespace `<agent>__<slug>` prevents collisions by construction), already reviewed (the Claude Code agent chose to memorize them as project-specific facts), and there is **no cross-agent clustering step** that could produce a false-positive merge needing human review. The entire purpose of `_inbox/` — staging clustering decisions for human validation — does not apply. Writing project files directly to `shared/project/` is the correct workflow, not an expedient shortcut.

**Relationship to the "plugin never writes to `shared/<type>/<slug>.md`" rule from §4.4:** that rule applies fully to `shared/feedback/`, `shared/user/`, and `shared/reference/`. `shared/project/` is a **deliberate carve-out**, documented as such in §4.4. The plugin-writable surface inside `shared/` is precisely `shared/_inbox/**` + `shared/project/**`.

**Collision handling:** two project files from the same agent with the same slug should not happen (Claude Code filenames are unique per agent), but if it does, `promote.py` raises `PromoteCollisionError` and the orchestrator logs and skips both.

### 6.6 — New: `core/extract/__init__.py` (~100 LOC)

The orchestrator. Public entry point for the whole pipeline.

```python
@dataclass
class ExtractionSummary:
    projects_promoted: int
    pages_written: int
    sibling_proposed: int
    update_proposed: int
    unchanged_skipped: int
    dismissed_skipped: int

**Sequence:**

```python
def run_extraction(cfg, *, dry_run=False, force=False):
    start = time.monotonic()
    vault_root = paths.vault_root(cfg)
    state_path = vault_root / ".mnemo" / "extraction-state.json"

    with locks.try_lock(vault_root / ".mnemo" / "extract.lock") as lock:
        if not lock.acquired:
            raise ExtractionIOError("another extraction is in progress")

        state = load_state(state_path, force=force)
        scan_result = scanner.scan(vault_root, state)

        if dry_run:
            return _print_estimate(scan_result, cfg)

        summary = ExtractionSummary()

        # Phase 1: projects (zero LLM, fastest, can't fail from network)
        project_result = promote.promote_projects(
            scan_result.by_type.get("project", []),
            state,
            vault_root,
        )
        summary.projects_promoted = len(project_result.written_fresh) + \
                                    len(project_result.overwrite_safe)
        inbox.atomic_write_state(state, state_path)

        # Phase 2+: LLM types
        for type_name in ("feedback", "user", "reference"):
            files = scan_result.by_type.get(type_name, [])
            if not files:
                continue

            all_pages = []
            for chunk in prompts.chunks_for(files, cfg["extraction"]["chunkSize"]):
                build = {
                    "feedback": prompts.build_feedback_prompt,
                    "user":     prompts.build_user_prompt,
                    "reference":prompts.build_reference_prompt,
                }[type_name]
                system = {
                    "feedback": prompts.FEEDBACK_SYSTEM_PROMPT,
                    "user":     prompts.USER_SYSTEM_PROMPT,
                    "reference":prompts.REFERENCE_SYSTEM_PROMPT,
                }[type_name]

                try:
                    response = llm.call(
                        build(chunk),
                        system=system,
                        model=cfg["extraction"]["model"],
                        timeout=cfg["extraction"]["subprocessTimeout"],
                    )
                except (llm.LLMSubprocessError, llm.LLMParseError) as exc:
                    errors.log_error(vault_root, "extract.chunk", exc)
                    summary.failed_chunks += 1
                    continue
                # KeyboardInterrupt, OSError, etc. propagate

                summary.llm_calls += 1
                summary.total_cost_usd += response.total_cost_usd or 0.0
                summary.total_input_tokens += response.input_tokens or 0
                summary.total_output_tokens += response.output_tokens or 0
                if response.api_key_source != "none":
                    summary.all_calls_subscription = False

                try:
                    payload = _extract_pages_from_response(response.text)
                except LLMParseError as exc:
                    errors.log_error(vault_root, "extract.chunk", exc)
                    summary.failed_chunks += 1
                    continue

                all_pages.extend(payload)

            # Dedupe across chunks within this type
            deduped = inbox.dedupe_by_slug(all_pages)
            apply_result = inbox.apply_pages(deduped, state, vault_root)
            _merge_apply_into_summary(apply_result, summary)
            inbox.atomic_write_state(state, state_path)

        state.last_run = datetime.now().isoformat(timespec="seconds")
        inbox.atomic_write_state(state, state_path)
        summary.wall_time_s = time.monotonic() - start
        return summary
```

**Explicit non-handling of `KeyboardInterrupt`:** the only `except` clauses use typed exceptions. `KeyboardInterrupt` (which is a `BaseException`, not `Exception`) propagates out naturally and kills the run. State from completed chunks is already flushed. Anti-patterns to avoid:

- ❌ `except Exception as exc:` — wrong signal to readers; easy to widen by accident.
- ❌ `except:` (bare) — catches `KeyboardInterrupt` and `SystemExit`. **Forbidden** in `core/extract/`.
- ✅ `except (LLMSubprocessError, LLMParseError) as exc:` — explicit, typed, intent-clear.

```python
def _maybe_emit_hint(cfg, vault_root):
    try:
        state_path = vault_root / ".mnemo" / "extraction-state.json"
        if not state_path.exists():
            return  # first-run semantics: no hint until first extract
        state = _load_state_best_effort(state_path)
        last_run = state.get("last_run")
        if not last_run:
            return
        last_run_ts = datetime.fromisoformat(last_run).timestamp()

        count = 0
        for p in (vault_root / "bots").glob("*/memory/*.md"):
            if p.name == "MEMORY.md":
                continue
            try:
                if p.stat().st_mtime > last_run_ts:
                    count += 1
            except OSError:
                continue

        threshold = cfg.get("extraction", {}).get("hintThreshold", 5)
        if count < threshold:
            return

        # Per-day dedup: check today's log for existing hint line
        today_log = _today_log_path(vault_root, agent_from_cache_or_cwd())
        if today_log.exists() and "🟡" in today_log.read_text(errors="ignore"):
            return

        if count >= threshold * 3:
            line = f"🟡 {count} new memories (a lot!) — run /mnemo extract"
        else:
            line = f"🟡 {count} new memories — run /mnemo extract"
        log_writer.append_line(agent, line, vault_root)
    except Exception as exc:
        errors.log_error(vault_root, "session_end.hint", exc)
        # Never re-raise. Hint is cosmetic.
```

**Rate limits:**
- Max 1 hint per day per agent (enforced by the "already present" check).
- No retry, no backoff — the next SessionEnd runs the full check again.

**Circuit breaker compliance:** if `errors.should_run(vault_root)` returned False at the top of `session_end.main()` (as it does in v0.1), the hint block is never reached. Same fuse as every other hook body.

## 8. Error handling and robustness

### 8.1 — Governing rule (unchanged from v0.1)

**Hooks never crash the Claude Code session.** The hint emitter is the only new hook code and is wrapped in `try/except Exception` with silent fallback — it is explicitly cosmetic.

### 8.2 — New rule for v0.2

**Manual commands may fail loudly.** `/mnemo extract` is user-invoked and may print errors to stderr with actionable messages. Silence would hide the failure and frustrate the user. This does not contradict v0.1's "silent" ethos because v0.1's ethos was about background/hook behavior, not user-initiated commands.

### 8.3 — Failure domain table

See §6.1 for the `core/llm.py` failure table. Additional domains:

**Filesystem (`core/extract/inbox.py`, `core/extract/promote.py`):**

| Failure | Action |
|---|---|
| `PermissionError` on `_inbox/` write | Raise `ExtractionIOError`; orchestrator catches → aborts run; state from already-completed chunks is preserved (already flushed) |
| Disk full | Same as above; atomic rename guarantees no partial file |
| Atomic rename fails (exotic FS) | Raise; user must investigate |

**State file (`~/mnemo/.mnemo/extraction-state.json`):**

| Failure | Action |
|---|---|
| JSON invalid on read | Rename existing file to `.bak.{timestamp}`, treat as empty state, log warning, run as fresh. Content-addressing still protects against overwriting user edits. |
| Unknown `schema_version` | Abort with clear error: "state file was written by a newer mnemo version" |
| Write fails (disk full, permission) | Run aborts with whatever state was last successfully flushed |

**Partial run semantics (explicit):**

Partial success is a valid final state, not a bug to rollback. If 5 chunks are planned and chunk 3 fails:
- Chunks 1, 2, 4, 5 results are on disk.
- Chunks 1, 2, 4, 5 state entries are flushed.
- Chunk 3 input files retain no state entry, remain "dirty" for next run.
- Exit code 1.
- Summary lists the failure explicitly.

### 8.4 — KeyboardInterrupt propagation

**Every `try/except` block in `core/extract/` uses typed exceptions only.**

- ✅ `except (LLMSubprocessError, LLMParseError) as exc:` — typed, explicit.
- ✅ `except OSError as exc:` — specific.
- ❌ `except Exception as exc:` — wrong signal, forbidden in `core/extract/`.
- ❌ `except:` (bare) — catches `KeyboardInterrupt` and `SystemExit`, **forbidden**.

`KeyboardInterrupt` (a `BaseException`, not `Exception`) therefore propagates freely. When the user hits Ctrl+C:

1. The current `llm.call()` subprocess receives SIGINT (propagated), dies.
2. `subprocess.run` raises `KeyboardInterrupt`.
3. The orchestrator's for-loop exits.
4. Python's cleanup runs; the `locks.try_lock` context manager releases the lock.
5. State is **whatever was last flushed** — which is chunk-granular, so at most the current chunk is lost.
6. Process exits with the standard SIGINT code.

No retries, no partial chunk writes, no "recovery" logic. The next `mnemo extract` run picks up the lost chunk naturally via the dirty-file diff.

### 8.5 — Concurrency: two `mnemo extract` runs simultaneously

Protected by `core/locks.try_lock(vault_root / ".mnemo" / "extract.lock")`. The second run aborts immediately with a clear message: "another extraction is in progress (held by PID X)". No queue, no wait — the user can retry manually once the first run completes.

Reuses the v0.1 atomic-mkdir lock primitive (cross-platform, no OS-specific imports). Self-heals against stale locks (>5min old).

### 8.6 — Circuit breaker interaction

`core/errors.py::should_run()` is modified (+~5 LOC) to filter out entries with `where` prefix `extract.*` when counting hourly errors. Rationale: a user running `mnemo extract` 12 times in a bad-internet hour should not disable the capture hooks that are v0.1's primary value. Extraction failures are the user's problem; hook failures are the plugin's problem.

### 8.7 — Explicitly not handled

- **LLM hallucinations that pass schema check.** Structural validation only. Semantic quality is validated by human review in `_inbox/`.
- **LLM writing malicious content into pages.** The LLM is the user's own Claude subscription running locally; the threat model is "low". Pages are plain markdown and are never executed.
- **Pathological memory file content that breaks prompt parsing.** The `<<<FILE:>>>` separator uses a character sequence unlikely in real markdown. If a memory file contains the literal separator, the LLM may get confused; we accept this as a non-issue unless real dogfood surfaces it.
- **Windows native (non-WSL).** Inherits v0.1's "best-effort" stance. Primary targets are Linux, macOS, WSL.

## 9. Testing strategy

### 9.1 — Layering (inherited from v0.1)

`tests/unit/` — fast, isolated, uses `tmp_path` and monkeypatches.
`tests/integration/` — full pipeline with mocked LLM.
`tests/e2e/` — real subprocess, opt-in, skipped by default.

**Governing principle for v0.2:** `core/llm.py::call()` is the **single seam** where the subprocess boundary is mocked. Tests above this seam never touch real subprocesses, never make network calls, never depend on `claude` CLI presence.

### 9.2 — Unit tests

**`tests/unit/test_llm.py`** — the most rigorous suite because `core/llm.py` is where format resilience lives.

Fixture: `MockCompletedProcess(stdout, stderr, returncode)`. Monkey-patches `subprocess.run` in the `core.llm` namespace.

Cases:
- Clean JSON response → correct `LLMResponse` with all fields.
- Fenced response (`` ```json\n{...}\n``` ``) → fence stripped.
- Prose-prefixed response (`"Here is the JSON: ```json\n{...}```"`) → first `{` found, `raw_decode` parses to end-of-object.
- Prolix Haiku response (`"Sure! Here is the extraction you requested:\n\n```json\n{...}```\n\nLet me know if you need more."`) → only the object extracted.
- Trailing garbage (`{...} extra commentary`) → `raw_decode` stops at correct position.
- Empty response → `LLMParseError("no JSON object found")`.
- Syntactically invalid JSON → `LLMParseError` with offset.
- `returncode != 0` + empty stderr → `LLMSubprocessError`.
- `returncode != 0` + stderr "API Error: rate limit exceeded" → 1 retry with backoff, eventual `LLMSubprocessError` if second attempt also fails.
- `subprocess.TimeoutExpired` → 1 retry with 2s backoff.
- `FileNotFoundError` → `LLMSubprocessError("claude CLI not found; install Claude Code first")`.
- `apiKeySource == "none"` → propagated to `LLMResponse.api_key_source`.
- `apiKeySource == "user"` → propagated.
- `total_cost_usd == 0.0` → propagated.
- `total_cost_usd` missing → `None` propagated (no crash).
- `usage.input_tokens` present → propagated.
- `usage` missing entirely → `None`s propagated.

**`tests/unit/test_extract_scanner.py`**
- Mini vault with 2 agents × 3 files → correct grouping.
- `MEMORY.md` skipped.
- Missing frontmatter → defaulted to `feedback` with warning.
- Unknown `type` → defaulted to `feedback` with warning.
- `source_hash` deterministic across runs.
- Diff against state: unchanged slugs correctly identified.
- Empty vault → empty `ScanResult`.

**`tests/unit/test_extract_prompts.py`**
- Each builder with 1, 3, 10, 15 files.
- Chunking: 15 files → chunks of [10, 5]; order preserved.
- Structural assertions only: `<<<FILE:` markers present, schema JSON present, few-shots present.
- No quality assertions (those are e2e / dogfood).

**`tests/unit/test_extract_inbox.py`** — the highest-coverage unit suite because this module holds the core invariant.

Every row of the §5.4 decision table gets a test:
- `fresh` write (entry absent).
- `unchanged` skip (source hash matches state).
- `overwrite_safe` (source changed, on-disk matches written_hash).
- `sibling_proposed` (source changed, on-disk ≠ written_hash).
- `promoted_updated` (entry in state, file absent from inbox, present in shared/).
- `dismissed` (entry in state, file absent from both).
- `dismissed_skip` (status already dismissed).
- `legacy_promoted` (status promoted, shared file gone).

Plus:
- Post-process dedupe: 2 chunks with same slug → merged `source_files`, body from larger cluster.

## 10. Open questions (for future phases)

v0.2 deliberately leaves these for later:

1. **Live diff review of promoted pages.** v0.2 writes `.update-proposed.md` siblings for already-promoted slugs with changed sources. v0.3 could introduce an interactive merge flow (`mnemo review`) or integrate with Obsidian via a plugin notification.
2. **Log-based extraction (v0.3).** When and how to layer daily logs on top of memory extraction — token budgeting, noise filtering, incremental semantics.
3. **Session transcript capture.** Whether v0.3 or v0.4 should start capturing `transcript_path` content for richer extraction inputs.
4. **Auto-promotion of high-confidence clusters.** If the LLM returns a confidence score per page, could we auto-promote ≥0.95 into `shared/<type>/` without review? Dangerous; likely no.
5. **Extraction quality telemetry.** A local-only `mnemo doctor --extraction` command that shows ratio of accepted vs dismissed pages over time, surface quality drift.
6. **Model upgrades.** When Haiku 5 ships, audit cost/quality tradeoff and update the default.

## 11. Design decisions log

Decisions made during the v0.2 brainstorm, preserved for future maintainers:

| # | Decision | Rationale |
|---|---|---|
| 17 | v0.2 scope = LLM extraction only (sync/backup/auto-update out of scope) | Focus; those questions are orthogonal to extraction and can evolve independently |
| 18 | Input source = mirrored memory files only, not daily logs | Memory is already in canonical format; logs are 80% noise and can come in v0.3 |
| 19 | Merge strategy = type-aware hybrid | `project` 1:1 (local by nature), `feedback`/`user`/`reference` cluster globally via LLM — mirrors the real semantics of each memory type |
| 20 | Trigger = manual `/mnemo extract` + passive hint in SessionEnd | Avoids running expensive global clustering on every SessionEnd; hint preserves the "invisible PKM" promise |
| 21 | Inbox semantics = content-addressable via state file | Preserves review work across re-extractions — the hard invariant |
| 22 | Execution = `claude --print` subprocess | Maintains v0.1 stdlib-only invariant; reuses subscription auth via the CLI; SDK adds nothing for this use case |
| 23 | Prompt shape = one LLM call per type (chunked ≤10 files/call) | Balance wall-time against per-chunk context budget; 10 is conservative for Haiku attention |
| 24 | Cost surfacing = pre-flight estimate + post-hoc real cost from CLI | Never hardcode prices; trust the `total_cost_usd` from the subprocess envelope; show tokens for subscription users |
| 25 | `shared/` layout change: mirror memory types, not v0.1 speculative taxonomy | v0.1's `people/`/`companies/`/`projects/`/`decisions/` was speculative and stayed empty; mirror the real types (`feedback`/`user`/`reference`/`project`) |
| 26 | `shared/project/` is plugin-writable; all others are `_inbox/`-only | Project files are 1:1 with no cluster risk; carving out an exception simplifies the common case |
| 27 | Extraction errors filtered from hook circuit breaker | Manual command failures must not disable capture hooks |
| 28 | `core/extract/` uses typed exceptions only (no `except Exception`) | KeyboardInterrupt must propagate; typed exceptions are intent-clear |
| 29 | Resilient JSON parsing via `raw_decode` | Haiku ignores "be terse" instructions; prose-prefixed output must parse without human intervention |
| 30 | Post-process dedupe by slug across chunks | When a type splits into multiple chunks, cross-chunk cluster collisions are resolved by merging source_files |

## 13. File formats

### 13.1 — Extracted page format

```markdown
---
name: No commits without permission
description: User-preference rule about never creating git commits without explicit approval
type: feedback
extracted_at: 2026-04-13T15:30:00
extraction_run: 2026-04-13T15:30:00-abc123
sources:
  - bots/central-inteligencia-frontend/memory/feedback_no_commits.md
  - bots/clubinho/memory/feedback_no_commit_without_permission.md
tags:
  - needs-review
---

Do not create git commits unless the user explicitly asks.

**Why:** The user has been burned by automated commits polluting history and wants manual control over the commit log.

**How to apply:** In every agent interaction, treat commit operations as explicit-consent-required actions. If the user's request implies a commit (e.g., "ship this fix"), confirm before running `git commit`.
```

The `sources` list lives in frontmatter only — the body is purely the merged canonical content, unpolluted by metadata.
