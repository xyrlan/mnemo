**Version:** v0.5.x design
**Date:** 2026-04-15
**Status:** Draft (awaiting implementation plan)
**Author:** xyrlan
**Predecessor:** [v0.5 MCP injection](2026-04-13-mnemo-v0.3-design.md) (shipped via PR #19) + [PR #22 PreToolUse rule activation](../../README.md#rule-activation-v05) (merged 2026-04-15)

---

## 1. Summary

v0.5.x closes a real efficiency gap in mnemo's **pull retrieval surface** (MCP) without
touching the **push retrieval surface** (PreToolUse). The gap surfaced during the
critical review of the MCP flow: the current two-step retrieval (`list_rules_by_topic`
→ `read_mnemo_rule`) forces Claude to play coarse-grained semantic match against a
9-tag ontology, costs 2+ round-trips per answer, has zero observability, and the
SessionStart injection merely lists topic names instead of surfacing high-value rules
directly.

Three coordinated features ship together under the existing `injection.enabled` flag,
forming a "pincer" over the retrieval problem:

**Feature A — `search_rules` MCP tool.** One-step retrieval with TF-IDF-lite scoring
over name+description+tags+body_preview. Replaces the `list → read` cycle for
specific queries, breaks the tag-ontology bottleneck, returns full bodies inline.
`scope="project"` default, `scope="vault"` opt-in.

**Feature B — `.mnemo/mcp-access-log.jsonl`.** Append-only JSONL capturing every MCP
tool call with query content, hit slugs, elapsed time. Motor for the feedback loop:
`mnemo doctor` gains two new checks (dead rules, recurrent zero-hit queries). Coexists
with the aggregate counter for statusline hot-path.

**Feature C — Rich SessionStart payload.** `_build_injection_payload` upgraded to
surface top 3 enrich rules of the current project with body previews inline, plus the
existing topic list with tags de-duplicated. Fallback to vault-wide top rules when
the current project has fewer than 3 rules, explicitly labeled as fallback.

**Cross-cutting invariant.** All four MCP tools (the three existing + `search_rules`)
gain a `scope="project"|"vault"` parameter, default `"project"`. Project is derived
per-call via `resolve_agent(str(Path.cwd())).name` with silent fallback to vault-wide
when derivation fails. This closes a pre-existing gap where legacy tools returned
vault-wide unions while the user's binding explicitly required per-project
segregation.

**Out of scope.** Semantic search via embeddings (deferred to v0.5.y if telemetry
shows that substring+TF-IDF is insufficient), reverse-lookup "rules citing file X",
telemetry-weighted ranking (deferred to v0.5.y after ~2 weeks of access-log data).
None of these are required for the pincer to work; all three would be net scope creep.

## 2. Problem and motivation

### 2.1 The pull MCP flow is not efficient today

The critical review on 2026-04-15 surfaced five concrete weaknesses in the v0.5 pull
retrieval flow:

1. **Retrieval is top-down via tag, when the query is fine-grained.** The vault has
   9 topic tags. A query like "should I use `useMemo` for this component?" touches
   `react` + `caching` — two tags, neither precise. The model guesses a tag, calls
   `list_rules_by_topic("react")`, gets a dump of 10 react rules most of which are
   irrelevant to the actual question, has to pick. Signal-to-noise is bad at the list
   step because 9 coarse buckets cannot discriminate semantically.

2. **Two-step retrieval burns turn budget.** Every rule read costs ≥2 MCP calls
   (`list_rules_by_topic` → `read_mnemo_rule`). For quick advisory questions, the
   model skips MCP entirely and answers from its own training — meaning the vault
   exists but doesn't get consulted. The cost structure is punishing against the very
   use case that justifies MCP.

3. **Zero observability.** `.mnemo/mcp-call-counter.json` tracks aggregate counts
   but doesn't distinguish which tags get used, which slugs get listed but never
   read, which queries produce zero hits. There's no feedback loop to improve the
   ontology.

4. **No recency or usage signal.** `list_rules_by_topic` orders by `source_count
   desc` — a 6-month-old rule with 5 sources ranks above a 2-day-old rule with 1
   source, even if the newer one is more currently relevant. Dead rules never leave
   the pool.

5. **SessionStart injection is thin.** It prints topic names only, forcing the
   model to translate "react" into "maybe there's a rule I should look up". A richer
   injection with the top rules' bodies preview would eliminate the cold-start call
   for high-value cases.

Separately, the legacy MCP tools (`list_rules_by_topic`, `read_mnemo_rule`,
`get_mnemo_topics`) return vault-wide unions — rules owned by project A surface when
Claude is working in project B. This contradicts the user's binding memory from
2026-04-15: *"per-project segregation is correct, do NOT add global scope"*. The gap
is pre-existing but becomes embarrassing when `search_rules` ships as project-
filtered by default.

### 2.2 What the push retrieval path (PR #22) already solved, and what it didn't

The PreToolUse hook from PR #22 pushes rules directly into Claude's turn at the tool-
call boundary — denying Bash commands matching `enforce.deny_pattern` and injecting
`additionalContext` for Edit/Write/MultiEdit on files matching `activates_on.
path_globs`. This is strictly better than pull for the cases where it applies: zero
model decision, zero retrieval latency, exact point-in-time injection.

But push only fires at tool boundaries. Conversational queries, architectural
discussions, and advisory questions ("what's our convention for X?") still depend on
pull. After PR #22, the bar for pull retrieval went up: its remaining use cases are
all the ones where the LLM needs to reason about the vault *before* touching code.
That reasoning is exactly what the current flow does worst.

### 2.3 Why a pincer attack

The three features compound:

- Feature C (rich SessionStart) attacks the **head of the distribution** — the
  handful of rules that dominate daily use in a given project already land in the
  model's context at zero incremental latency.
- Feature A (`search_rules`) attacks the **long tail** — for niche or specific
  queries, one call returns full bodies inline instead of list+read.
- Feature B (access log) **measures the whole loop** so the next iteration can swap
  the source_count ranking heuristic in Feature C for a telemetry-weighted one.

Together they turn pull MCP from a "call this if you remember" feature into a
default-useful surface that sees real traffic.

## 3. Scope and non-goals

### 3.1 In scope for v0.5.x

- New MCP tool `search_rules(query, scope, limit)` with hand-rolled TF-IDF-lite
  scoring (no new dependencies)
- New observability module `core/mcp/access_log.py` writing JSONL
- Rich SessionStart payload with Top 3 enrich rules + deduplicated topic list
- Project filter retroactively applied to all existing MCP tools via `scope` parameter
- New DEFAULTS sub-flag `injection.telemetry.enabled: true`
- New `core/log_utils.py` module extracting `rotate_if_needed` for shared use
- New `core/rule_activation.py` public function `get_top_enrich_rules` returning
  `TopEnrichResult` dataclass
- New `cmd_doctor` checks: dead rules (listed ≥10× but never read) and recurrent
  zero-hit queries
- Updated tool descriptions in MCP server's `_TOOL_DEFS` documenting `scope` parameter

### 3.2 Out of scope (explicit deferrals)

- **Semantic search via embeddings.** Would require a small embedding model
  dependency, breaking the `dependencies = []` invariant. Deferred to v0.5.y only if
  access-log data shows substring+TF-IDF routinely fails on long-tail queries.
- **Telemetry-weighted ranking in Feature C.** Requires ≥2 weeks of access-log data
  to be meaningful. A dedicated follow-up ticket will be filed at implementation
  time with a clear entry criterion ("N days of `mcp-access-log.jsonl` data
  available"). Feature C ships with source_count as explicit v1 placeholder.
- **Reverse-lookup "rules citing file X".** Different problem shape; partially
  addressed by the PreToolUse enrich path already. Deferred indefinitely unless a
  specific use case emerges.
- **Deep tag hierarchy** (`react.hooks.caching`). Violates controlled-vocabulary
  binding; low ROI.
- **LLM-powered result re-ranking.** Adds latency and tokens for what
  TF-IDF-lite solves cheaply.
- **Per-project context threading in the MCP protocol** (via `initialize` params or
  explicit session IDs). Deferred in favor of `Path.cwd()` per-call derivation.

### 3.3 Invariants preserved

- `dependencies = []` — all four new code additions are pure stdlib (`re`, `json`,
  `pathlib`, `time`, `datetime`).
- Hand-rolled parsing continues — no new YAML / TOML / search library.
- Fail-open absolute — no new code path propagates exceptions to Claude Code.
- Per-project segregation — all user-visible retrieval is project-scoped by default,
  with explicit opt-out via `scope="vault"`.
- Auto-populating over manual curation — no new user commands, no new config flags
  for day-to-day operation, telemetry is observability only.

## 4. Design

### 4.1 Architecture overview

Three features share the per-project filter and the MCP server as cross-cutting
infrastructure. Control flow for a single pull retrieval:

```
[Claude] ──JSON-RPC──▶ [mnemo mcp-server]
                          │
                          ├── tools/call received
                          │
                          ├── project = _resolve_current_project(vault_root)
                          │     ├── Path.cwd() → resolve_agent(cwd).name
                          │     └── try/except → None on failure (silent fallback)
                          │
                          ├── scope_effective = project ? args.scope : "vault"
                          │
                          ├── Dispatch to core/mcp/tools:
                          │     ├── search_rules(vault_root, query, scope=..., project=...)
                          │     ├── list_rules_by_topic(vault_root, topic, scope=..., project=...)
                          │     ├── read_mnemo_rule(vault_root, slug, scope=..., project=...)
                          │     └── get_mnemo_topics(vault_root, scope=..., project=...)
                          │
                          ├── counter.increment(vault_root)   [existing, hot path]
                          ├── access_log.record(vault_root, {...}) [new, if telemetry]
                          │
                          └── return _ok(req_id, _text_content(result))
```

Session start retrieval for rich payload (pre-loading high-value rules):

```
[Claude startup] ──stdin JSON──▶ [session_start hook]
                                    │
                                    ├── config, vault, circuit breaker (existing)
                                    ├── injection.enabled gate (existing)
                                    │
                                    ├── current_project = resolve_agent(cwd).name
                                    ├── build_index → write_index (if flags on)
                                    ├── index = load_index(vault)
                                    │
                                    ├── payload_text = _build_rich_injection_payload(
                                    │                     vault, current_project, index)
                                    │     │
                                    │     ├── top_result = rule_activation.get_top_enrich_rules(
                                    │     │                   index, current_project, limit=3)
                                    │     │     ├── Project rules by source_count desc, slug asc
                                    │     │     ├── Fallback vault-wide if <limit
                                    │     │     └── Returns TopEnrichResult dataclass
                                    │     │
                                    │     ├── Header + Top 3 section (labeled fallback distinctly)
                                    │     ├── Topic list (project-filtered, dedup against Top 3)
                                    │     └── on_any_exception → fall back to legacy payload
                                    │
                                    └── _emit_injection(payload_text) → stdout
```

### 4.2 Components

#### 4.2.1 New files (2 code + 4 test files)

**`src/mnemo/core/log_utils.py`** (~20 LOC)

Extracted from `rule_activation.py`. Single public function `rotate_if_needed(path:
Path, max_bytes: int) -> None`. One-level rotation (overwrites existing `.1`).
Silent on all errors. Consumed by `rule_activation.log_denial`,
`rule_activation.log_enrichment`, and the new `access_log.record`.

**`src/mnemo/core/mcp/access_log.py`** (~80 LOC)

Public function `record(vault_root: Path, entry: dict) -> None`. Reads
`injection.telemetry.enabled` from config; returns immediately if False. Calls
`rotate_if_needed`, creates `.mnemo/` if missing, appends one JSON line to
`mcp-access-log.jsonl`, flushes. Silent on all errors. Never blocks the tool call.

**`tests/unit/test_log_utils.py`** (~5 tests) — rotation noop, cap trigger,
existing-rotation overwrite, missing file handling, fail-safe on OSError.

**`tests/unit/test_mcp_access_log.py`** (~8 tests) — append shape, telemetry flag
respect, rotation integration, non-serializable entry safety, parent dir creation,
unwritable vault fail-open.

**`tests/unit/test_mcp_search_rules.py`** (~25 tests in 3 blocks: tokenization,
scoring, full search behavior)

**`tests/unit/test_mcp_tools_project_filter.py`** (~12 tests) — `scope="project"|"vault"`
behavior on all four MCP tools, project derivation fallback.

**`tests/integration/test_mcp_retrieval_e2e.py`** (~8 tests) — full JSON-RPC dispatch,
counter + access log integration, project derivation per-call, telemetry flag gating,
SessionStart rich payload full loop.

#### 4.2.2 Modified files (5)

**`src/mnemo/core/mcp/tools.py`**

Largest delta. Additions:

- `search_rules(vault_root, query, *, scope="project", project=None, limit=3) ->
  list[dict]` — main new entry point.
- Private helpers:
  - `_tokenize_query(q)` — case-fold, word-boundary split via `re.findall(r"\w+",
    q)`, drop words <3 chars, drop stopwords from a small hardcoded English/PT list
    (`the, and, for, are, was, with, that, this, how, should, when, what, why,
    where, which, para, com, como, quando, qual, onde`, ~25 entries), return
    `list[str]`. The stopword list is intentionally tiny: we are de-noising
    questions, not doing information retrieval research.
  - `_score_rule(tokens, fm, body)` — multi-field scoring with weights:
    slug=10, name=5, tag=3, description=2, body=1. Returns int.
  - `_resolve_current_project(vault_root)` — `Path.cwd()` → `resolve_agent` → name.
    Returns `None` on any exception; logs to `.errors.log`.
  - `_rule_belongs_to_project(fm, project)` — checks `sources:` prefix `bots/<project>/`.
- New `scope: str = "project"` parameter on `list_rules_by_topic`, `read_mnemo_rule`,
  `get_mnemo_topics`. Also accepts `project: str | None = None` for explicit injection
  by the server; falls back to `_resolve_current_project` when None.
- No class / API break — all additions, existing signatures still work via default
  parameters.

Ranking within `search_rules`: primary by computed score descending, then
`source_count` desc, then `slug` asc for deterministic tiebreak. Threshold check:
any rule with score `< 3` is excluded from the result set entirely (prevents noise
from a rule hitting a single body word). If all candidates fall below threshold,
return empty list. This floor of 3 corresponds to "at least one tag hit" or
"at least a name partial match" under the weight table.

**`src/mnemo/core/mcp/server.py`**

Delta:

- New `search_rules` entry in `_TOOL_DEFS` with JSON inputSchema:
  `{query: string, scope?: string (project|vault), limit?: integer}`.
- Updated descriptions on the three legacy tool entries mentioning the `scope`
  parameter and how to use `scope="vault"` for cross-project lookups.
- `_handle_tool_call` resolves `project` once per call via
  `mnemo.core.mcp.tools._resolve_current_project(vault_root)`.
- Dispatches to tool functions passing `project=project` and honoring the user-
  supplied `args["scope"]` with default `"project"`.
- After each successful dispatch, calls both `counter.increment(vault_root)` and
  `access_log.record(vault_root, entry)` — the latter wrapped so telemetry flag off
  means zero writes.
- Measures `elapsed_ms` via `time.perf_counter()` around the dispatch for the log
  entry.

**`src/mnemo/core/rule_activation.py`**

Additions:

- New `@dataclass(frozen=True) TopEnrichResult` with fields `project_rules: tuple[dict, ...]`,
  `fallback_rules: tuple[dict, ...]`, `had_fallback: bool`.
- New public function `get_top_enrich_rules(index: dict, current_project: str, *,
  limit: int = 3) -> TopEnrichResult`. Logic:
  1. Sort `index["enrich_by_project"][current_project]` by
     `(-source_count, slug)`, take first `limit`.
  2. If fewer than `limit` selected, walk `enrich_by_project` for other projects,
     flatten, exclude already-selected slugs (dedup), sort, take gap-filler slots.
  3. Return `TopEnrichResult` with `had_fallback=True` if any fallback rules were
     used.
- Refactor: `_rotate_if_needed` (private) is removed from this file; existing
  `log_denial` and `log_enrichment` import `rotate_if_needed` from
  `mnemo.core.log_utils`. Pure refactor, no behavior change.

**`src/mnemo/hooks/session_start.py`**

Delta in `_build_injection_payload`:

- Calls `rule_activation.get_top_enrich_rules(index, current_project, limit=3)`.
- Builds the rich text in sections:
  - **Header**: "You have access to the mnemo project brain via MCP. Use
    `search_rules(query)` for specific questions..."
  - **Top 3 section**: one bullet per rule, format `• [[<slug>]]: <body_preview>`
    for project rules. For fallback rules, append `(vault-wide fallback)` suffix to
    the bullet. Section header mentions the heuristic explicitly: "Top active rules
    (v1 heuristic: source_count)".
  - **Topic list section**: "Other known topics:" followed by comma-separated tags
    from `index["enrich_by_project"][current_project]` topic_tags, minus the tags
    already covered by the Top 3 rules.
- Wraps the entire rich build in try/except. On any failure, falls back to
  `_build_legacy_injection_payload(vault_root, current_project)` — the topic-list-
  only payload from PR #22 Task #6.
- When `get_top_enrich_rules` returns fully empty (project empty AND vault empty),
  also falls back to legacy payload. No empty Top 3 section with silence.

**`src/mnemo/core/config.py`**

Adds `injection.telemetry.enabled: True` under the existing `injection` block in
DEFAULTS. No other keys changed.

**`src/mnemo/cli.py`**

`cmd_doctor` gains `_doctor_check_mcp_access_log(vault)`:

- Reads last ~30 days of `mcp-access-log.jsonl` (capped scan at 10k lines for
  safety).
- **Dead rules check**: groups entries by `hit_slugs`, counts how many times each
  slug appeared in `list_rules_by_topic` or `search_rules` results. For any slug
  with count ≥10 and zero `read_mnemo_rule` events: emit warning "⚠ Rule `<slug>`
  was listed N times but never read — consider rewriting its description or tags."
- **Recurrent zero-hit queries**: filter entries with `tool == "search_rules"` and
  `result_count == 0`. Cluster by substring similarity (simple bucketing by first
  3 tokens). For clusters with size ≥3: emit warning "⚠ Queries like `<example>`
  returned zero hits N times — consider authoring a rule for this topic."

Both checks are silent when the access log is empty, missing, or telemetry is off.

### 4.3 Data flow

See the ASCII control flow in §4.1 for both the MCP tool call and the
SessionStart rich payload.

Critical specifics:

**Project resolution per call.** The `_resolve_current_project` helper is invoked
once at the top of `_handle_tool_call`. Result is passed as `project=project` to all
tool functions, alongside the user-supplied `scope`. This means even if Claude Code
somehow reuses a server process across workspace switches (which would be a protocol
violation but isn't forbidden), each call independently reflects the current working
directory. One syscall (`Path.cwd()`) + one tree walk (`resolve_agent`) per call.
Cost: microseconds.

**Scope effective.** The server resolves the "effective" scope as follows:

```
if project is None and args.scope == "project":
    scope_effective = "vault"   # silent fallback
else:
    scope_effective = args.scope  # default "project"
```

The caller never needs to know about the fallback. Tools receive `scope_effective`
as an explicit string.

**Access log write.** After every successful tool dispatch (or failed dispatch that
returned an empty result — still worth logging), the server assembles an entry:

```python
{
    "timestamp": "<ISO-8601-Z>",
    "tool": "<search_rules|list_rules_by_topic|read_mnemo_rule|get_mnemo_topics>",
    "args": {<sanitized>},        # query strings yes, large bodies/results no
    "scope_requested": "project|vault",
    "scope_effective": "project|vault",
    "project": "<name>|null",
    "result_count": <int>,
    "hit_slugs": [<list>],         # empty for get_mnemo_topics
    "elapsed_ms": <float>,
}
```

Argument sanitization: keep query string verbatim (Q3 decided raw), truncate any
single string value longer than 1024 chars with a `…[truncated]` suffix, and drop
any key whose value is a list/dict larger than 10 elements. Result fields are
derived from the return value by introspection, not passed explicitly by the caller.

**SessionStart rich payload**. When the injection flag is on AND the index loaded
successfully AND the rich builder doesn't raise, the payload is the new rich format.
All other branches fall back to the legacy topic-list payload without warning —
zero regression surface.

### 4.4 Error handling (fail-open contract)

Four new surfaces, all wrapped in try/except with the existing patterns:

| Surface | Failure mode | Fallback behavior |
|---|---|---|
| `search_rules` outer | Unexpected exception | Return `[]`; no raise |
| `search_rules` per-rule loop | Malformed rule file | `continue`; skip one, keep scoring others |
| `_tokenize_query` | None / non-string input | Return `[]`; score loop no-ops |
| `_resolve_current_project` | resolve_agent raises | Return `None`; server applies scope=vault fallback |
| `_score_rule` | Missing fm fields | Score 0 for that field, continue |
| `get_top_enrich_rules` | Malformed index | Return empty `TopEnrichResult` |
| `_build_rich_injection_payload` | Any exception | Call `_build_legacy_injection_payload` |
| `access_log.record` | Anything (disk full, permission, serialization) | Silent pass |
| `rotate_if_needed` | OSError on rename | Silent pass (rotation is best-effort) |

**Deliberate non-integration of circuit breaker.** The `errors.should_run(vault)`
check is NOT applied to MCP tool calls. Rationale: MCP is a persistent service; if
the circuit breaker disabled all calls silently, Claude would receive empty results
every time after 10 errors in an hour without any indication mnemo was off. Worse UX
than the occasional error-logged failure. The `session_start` hook already has the
circuit breaker check at the top of `main()`, so the rich payload builder inherits
it transitively — this is sufficient.

## 5. Rollout

### 5.1 Ordering

The sprint is a single PR with roughly 5-6 commits in the following order (this maps
to the natural file-level dependency chain, not strict sequential work):

1. **`core/log_utils.py` extraction** — pure refactor, zero behavior change. Tests
   + updated imports in `rule_activation.py`. Verifies `rotate_if_needed` tests still
   pass against the new location.
2. **`core/mcp/access_log.py`** — new module with writer + tests. Depends on (1).
   Can be tested in isolation without touching the server.
3. **`core/mcp/tools.py` — `search_rules` + scope parameter on legacy tools** —
   the largest commit. Adds the new tool, adds the scope plumbing on the three
   existing tools, adds private helpers. Depends on nothing new.
4. **`core/rule_activation.py` — `get_top_enrich_rules`** — additive. Adds the
   `TopEnrichResult` dataclass and the ranking/fallback logic. Depends on nothing
   new.
5. **`core/mcp/server.py`** — wires everything together. Registers `search_rules`
   in `_TOOL_DEFS`, dispatches with project resolution, writes access log
   alongside counter. Depends on (2), (3).
6. **`hooks/session_start.py`** — calls `get_top_enrich_rules`, builds rich
   payload, legacy fallback. Depends on (4).
7. **`cli.py` — new doctor checks** — depends on access log data shape (2)
   being fixed.
8. **`core/config.py`** — DEFAULTS sub-flag. Trivial, can land first or last.

### 5.2 Activation

The features gate on existing flags:

- `injection.enabled = True` (already on in author's vault) gates Features A, B, C
  together (rich SessionStart, search_rules availability, project filter).
- New sub-flag `injection.telemetry.enabled = True` (default) gates only the access
  log writer (Feature B data collection). Users can disable telemetry while keeping
  the retrieval features.
- No other flags change. No new top-level flags.

### 5.3 Backward compatibility

- **Legacy MCP tool callers** (anyone calling `list_rules_by_topic("react")` without
  `scope`) will now receive project-scoped results by default. This is a behavioral
  change visible to Claude. Documented in tool descriptions; tests include a
  `scope="vault"` regression case reproducing the old behavior to prove opt-out works.
- **Legacy SessionStart injection consumers** (the invisible prompt Claude reads at
  session start) receive the rich payload. This is prose, not a structured contract,
  so there is no breaking API surface. The legacy topic-list-only payload is still
  reachable via the try/except fallback and via `injection.enabled = False`.
- **Existing `.mnemo/mcp-call-counter.json`** is unchanged — still the source of
  truth for the statusline. Access log is additive.

## 8. Glossary

- **Pull path** — retrieval initiated by Claude via MCP tool call. Model-driven,
  semantic-triggered.
- **Push path** — retrieval initiated by mnemo via PreToolUse hook at the tool-call
  boundary. Regex/glob-triggered, model-agnostic.
- **Head** of the retrieval distribution — high-frequency rules that dominate daily
  use in a given project. Target for Feature C (SessionStart rich injection).
- **Tail** of the retrieval distribution — specific, niche, lower-frequency
  queries. Target for Feature A (`search_rules`).
- **Feedback loop** — data pipeline from access log to future improvements
  (ranking, tag quality). Fueled by Feature B.
- **TF-IDF-lite** — multi-field substring scoring with hand-rolled tokenization and
  fixed weights. Not true TF-IDF (no document frequency weighting); a simple
  additive scorer that costs ~30 lines of Python and solves the obvious cases.
