# mnemo v0.3 — Auto-brain Mode Design

**Version:** v0.3 design
**Date:** 2026-04-13
**Status:** Draft (awaiting implementation plan)
**Author:** xyrlan
**Predecessor:** [v0.2 design](2026-04-13-mnemo-v0.2-design.md) (shipped to master, dogfood-validated)
**Hard prerequisite:** v0.2.1 (issues #5, #6, #7) — see §3.3.

---

## 1. Summary

v0.3 turns mnemo into the "auto-populating second brain" promised by the tagline. The v0.2 workflow — `mnemo extract` as a manual command, then manual promotion from `_inbox/` to `shared/` — shipped and worked, but the dogfood surfaced the obvious: **if the user has to run a command and move files, the plugin is just one more tool to maintain**. v0.3 closes that gap along two axes while preserving every v0.2 invariant.

**Eixo A — Background extraction.** The `SessionEnd` hook schedules extraction automatically via a fire-and-forget detached subprocess when debounce conditions are met. Zero manual trigger required in the common case. Hook runtime stays under 100ms.

**Eixo B — Auto-promote direct to `shared/<type>/`.** When the LLM produces a **single-source** page (one source file, zero clustering decisions), it is written directly to the sacred dir with `auto_promoted` tracking. **Multi-source** pages (cross-agent clusters where the LLM made a merge decision) continue to stage in `shared/_inbox/<type>/` for review. This split aligns the review surface with the decision surface.

**Scope boundary:** v0.3 is *only* auto-brain mode (Eixos A + B). Log-based extraction, session transcripts, interactive merge tooling, and model upgrades all remain deferred.

**Execution:** same `claude --print` subprocess pipeline as v0.2. Zero new runtime dependencies. The `Popen` detach mechanism is stdlib-only (`start_new_session` on POSIX, `creationflags` on Windows).

**Opt-in default.** `extraction.auto.enabled=false` on v0.3 ship. Users flip it on consciously after `mnemo doctor` points to it. v0.4 will consider flipping the default after sustained dogfood.

**Tagline reaffirmed:** "The Obsidian that populates itself so your Claude never forgets" — v0.2 made extraction tractable, v0.3 is where "populates itself" stops needing a human finger on the trigger.

## 2. Problem and motivation

### 2.1 — What the v0.2 dogfood showed

After v0.2 shipped on master (commit `ceb1cad`), the author ran `mnemo extract` once on the real `~/mnemo` vault and reviewed the output in `shared/_inbox/feedback/`:

- **The extraction pipeline worked.** 7 feedback files + 3 project files produced 7 feedback pages in `_inbox/` and 3 directly in `shared/project/` via the 1:1 promote path. The content-addressing, the resilient JSON parsing, the partial-failure semantics, the subprocess auth contract — all held up on the first real run.
- **The workflow was the bottleneck.** The user still had to (a) remember to run the command, (b) open each of the 7 files in Obsidian to review them, (c) hand-move the good ones to `shared/feedback/`, (d) think about whether duplicates should be merged. In practice, this review step happened once — out of pure curiosity — and was not going to happen a second time without friction.
- **Cross-agent clustering missed the clearest case.** Two feedback files from different agents expressed the same rule in different words:
  - `bots/central-inteligencia-frontend/memory/feedback_no_commits.md` ("don't create git commits")
  - `bots/clubinho/memory/feedback_no_commit_without_permission.md` ("never commit without explicit permission")

  The v0.2 prompt kept them separate. The whole point of cluster-merging is to catch exactly this pattern, and the few-shot examples in `prompts.py` were not strong enough to force the merge.

Together, these observations frame v0.3 as **two converging pressures**: automate the trigger (no more manual commands) and automate the promotion (no more hand-moving files). Both become safe only once the prompt quality reaches the point where the LLM merges duplicates reliably — which is the v0.2.1 prerequisite.

### 2.2 — Why auto-promote without a quality gate is worse than manual review

Auto-promote in v0.3 writes LLM output directly to `shared/<type>/`, which the user regards as their curated second brain. If the LLM emits two pages that should've been one (the `no-commits` vs `no-commit-without-permission` case), the user now has duplicates **in their sacred layer**. The cleanup cost is no longer "move one file from `_inbox/`" — it's "merge two canonical pages I was about to trust". That's strictly worse UX than v0.2.

The fix is to land v0.2.1 (prompt fixes + tools suppression + disable extended thinking) first, re-dogfood the manual flow to verify the same 7 files now produce ~5 pages with the cross-agent merge actually firing, **and only then** flip auto-brain on. The v0.3 spec encodes this as a hard gate, not a polite suggestion.

### 2.3 — Why split single-source from multi-source instead of all-or-nothing

Two alternatives were considered and rejected (see §11 decisions #31, #32):

1. **All-direct.** Every extracted page goes to `shared/<type>/` without staging. Simplest, but puts every LLM decision — including risky merge calls — into the sacred layer on first contact. A single bad merge pollutes the curated space.
2. **All-staged.** Every page stays in `_inbox/`. Safest, but this is literally v0.2 with a cron trigger — it doesn't deliver the "populates itself" promise.

The split by `len(source_files)` aligns the review surface with the accountability surface:

- **1 source → auto-promote.** The plugin made no editorial decision; it reformatted a file that Claude Code already chose to persist. Trust is inherited from capture.
- **≥2 sources → stage in `_inbox/`.** The plugin made a merge decision. Merge decisions deserve human validation. This is exactly the risky case.

Under v0.2.1's fixed prompt, most single-source pages are "Claude already memorized this fact, plugin just normalized the format" (low risk). Multi-source pages are "plugin decided these two files represent the same rule" (high risk, high value). The split keeps both sides honest.

## 3. Goals and non-goals

### 3.1 — Goals (v0.3)

1. **Zero manual trigger in the happy path.** The user never runs `mnemo extract` under normal operation. Background extraction fires on `SessionEnd` when debounce conditions are met.
2. **Hooks-never-raise preserved.** The new `SessionEnd` logic adds <100ms to the hook runtime, returns cleanly on every error path, and never blocks the session.
3. **Single-source pages auto-promote to `shared/<type>/`.** Content-addressable user-edit protection extends to the sacred layer — a conflict with user edits writes a `.proposed.md` sibling back into `_inbox/`, never overwrites the user's canonical copy.
4. **Multi-source pages stay in `_inbox/`.** The v0.2 review workflow is preserved for the decision-dense cases.
5. **Opt-in on first ship.** `extraction.auto.enabled` defaults to `false`. No silent LLM calls until the user explicitly opts in.
6. **Preserve every v0.1 and v0.2 invariant.** Hooks-only, stdlib-only, cross-platform, local-first, private, non-invasive, robust-by-default, Obsidian-optimal.
7. **Observability via `mnemo status` and `mnemo doctor`.** The user can see at a glance when the last auto-run happened, whether it succeeded, and whether one is currently running.

### 3.2 — Non-goals (v0.3)

1. **No daemons.** No persistent watcher, no cron, no systemd/launchd/Windows service. The only trigger is `SessionEnd`.
2. **No interactive merge tooling.** `.proposed.md` siblings are still plain files the user diffs in Obsidian or their editor of choice. A `mnemo review` command is deferred.
3. **No log-based extraction.** Daily logs still stay out of scope. The v0.2 memory-only input continues.
4. **No auto-flip of `auto.enabled` to `true`.** v0.3 ships with the default off. A future release will consider the flip after real-use telemetry.
5. **No semantic quality validation.** LLM hallucinations that pass schema check are still caught by human review of `_inbox/` (for multi-source) or by the content-addressing invariant when the user edits `shared/<type>/` pages.
6. **No new config key for "dismissed on shared/" entries.** The existing `dismissed` status machinery handles deletion of files from both `_inbox/` and `shared/<type>/` uniformly.
7. **No telemetry on acceptance ratio.** A future `mnemo doctor --extraction` that shows "auto-promoted / dismissed ratio over 30 days" is interesting but deferred to v0.4.

### 3.3 — Hard prerequisite: v0.2.1 must ship before `auto.enabled` is safe to flip

v0.2.1 is a separate release, described by GitHub issues **#5** (cross-agent merge prompt fix), **#6** (disable MCP plugin tools in the subprocess), and **#7** (disable extended thinking on Haiku 4.5). These fixes are orthogonal to the v0.3 architecture and ship on their own timeline. However, **v0.3's auto-brain machinery must not be activated until v0.2.1 has landed and been re-dogfooded manually**:

- The auto-promote path writes LLM output to the sacred layer. Bad clustering pollutes that layer directly.
- Extended thinking on every background run costs ~30 extra seconds per extraction, which breaks the 60-minute debounce floor in the tight case (multiple sessions per hour).
- MCP tool schemas loaded by default waste ~500-1000 input tokens per call, which multiplies over the 4-8 daily runs expected under v0.3.

**Encoding the gate in the spec:** v0.3 ships with `auto.enabled=false` by default and with v0.2.1's fixes already merged to master (release-process invariant: no v0.3 tag cut without v0.2.1 commits in the tree). The release-milestone gate is described in §12 step 8 — "Dogfood preparation: author validates v0.2.1 clustering on the real vault before flipping `auto.enabled=true`". There is no runtime check in `mnemo doctor` — see §6.8 for rationale.

## 4. Architecture

### 4.1 — High-level pipeline (v0.3 delta)

```
SessionEnd hook fires
   │
   ▼
existing v0.1/v0.2 steps (mirror, log line, cache clear, circuit breaker check)
   │
   ▼
_maybe_schedule_extraction(cfg, vault_root, agent_name):
    if not cfg.extraction.auto.enabled:
        fallback: _maybe_emit_hint()   # v0.2 passive hint behavior
        return
    load extraction-state.json (silent no-op on failure)
    count = files in bots/*/memory/*.md with mtime > state.last_run
    elapsed = now - state.last_run
    if count < auto.minNewMemories OR elapsed < auto.minIntervalMinutes:
        return
    if extract.lock is held:
        return  # another run already scheduled or in progress
    spawn_detached([sys.executable, "-m", "mnemo", "extract", "--background"])
    return  # hook returns in <100ms
   │
   ▼
(detached subprocess, independent lifetime)
   │
   ▼
run_extraction(cfg, background=True):
    acquire extract.lock
    scan, promote projects, extract cluster types (unchanged from v0.2)
    inbox.apply_pages decides per-page:
        len(source_files) == 1 → target = shared/<type>/<slug>.md
        len(source_files) >= 2 → target = shared/_inbox/<type>/<slug>.md
    same decision table §5.4 runs on each target
    persist last-auto-run.json (atomic)
    release lock
```

### 4.2 — Invariants preserved from v0.1 and v0.2

Every invariant from v0.1 (§4.3 of v0.1 design) and v0.2 (§4.3 of v0.2 design) carries forward unchanged. The only invariants that gain new text are:

- **Hooks-only.** The hook body gains a `Popen` spawn point, but the spawned process is detached — the hook itself does no work beyond the spawn decision and returns immediately.
- **`shared/<type>/` sacredness.** v0.2 carved out `shared/project/**` as plugin-writable. v0.3 extends the carveout: any markdown file in `shared/feedback/**`, `shared/user/**`, `shared/reference/**` whose frontmatter contains the `auto-promoted` tag may be rewritten by the plugin under the content-addressing rules. Files without that tag — including files the user promoted manually from `_inbox/` — remain untouchable.

### 4.3 — Invariants introduced by v0.3

- **Auto-promote is source-count gated.** A page with `len(source_files) == 1` is eligible for auto-promote. Any page with `len(source_files) >= 2` stages in `_inbox/`, regardless of any other signal (LLM confidence, source file age, agent identity). This is a hard rule with no override flag in v0.3.
- **Conflict siblings always land in `_inbox/`.** When the content-addressing rule fires for an auto-promoted page (user edited the sacred copy, source changed), the `.proposed.md` sibling is written to `shared/_inbox/<type>/<slug>.proposed.md` — never next to the sacred file. `_inbox/` is the unified review surface for all pending work.
- **`last_run` advances on success only.** Consistent with v0.2. Transient failures re-fire on the next debounce tick; persistent failures are surfaced via `mnemo doctor` and `last-auto-run.json`.
- **Lock semantics unchanged.** `extract.lock` serializes all extractions — background + manual + concurrent backgrounds. A scheduled background run that cannot acquire the lock exits silently. A manual `mnemo extract` run that cannot acquire the lock prints the existing "held by PID X" error from v0.2.
- **One-way transitions only.** `_inbox/ → shared/<type>/` is a one-way promotion. A page that was already auto-promoted never regresses to `_inbox/` (see §5.6 for the "single → multi upgrade" case).

### 4.4 — `shared/` layout change from v0.2

The layout itself does not change. What changes is **who writes to which paths**:

```
~/mnemo/shared/
├── _inbox/                      ← plugin-writable (v0.2 + v0.3)
│   ├── feedback/                   ← multi-source clusters + sibling .proposed.md
│   ├── user/
│   └── reference/
├── feedback/                    ← user-promoted + plugin-written auto-promoted (v0.3)
├── user/
├── reference/
├── project/                     ← 1:1 promoted (v0.2)
│   └── <agent>__<slug>.md
└── people.md                    ← legacy v0.1
```

The cluster-type directories (`feedback/`, `user/`, `reference/`) now contain a **mix** of two file classes, distinguished by frontmatter:

1. **User-promoted pages.** Frontmatter has no `auto-promoted` tag. Plugin never touches these.
2. **Plugin-auto-promoted pages.** Frontmatter has the `auto-promoted` tag plus `last_sync` timestamp. Plugin rewrites these under the content-addressing rules (overwrite if user has not edited them, sibling `.proposed.md` in `_inbox/` if they have).

The distinction is a runtime check — `_is_auto_promoted(path)` reads the frontmatter on every apply. There is no separate directory for the two classes.

## 5. Data model and decision tables

### 5.1 — State schema v2

Location: `~/mnemo/.mnemo/extraction-state.json` (unchanged).

```json
{
  "schema_version": 2,
  "last_run": "2026-04-13T15:30:00",
  "entries": {
    "feedback/use-yarn": {
      "source_files": ["bots/clubinho/memory/feedback_use_yarn.md"],
      "source_hash": "sha256:abc...",
      "written_hash": "sha256:def...",
      "written_at": "2026-04-13T15:30:00",
      "last_sync": "2026-04-13T15:30:00",
      "status": "auto_promoted"
    },
    "feedback/no-commits-without-permission": {
      "source_files": [
        "bots/central-inteligencia-frontend/memory/feedback_no_commits.md",
        "bots/clubinho/memory/feedback_no_commit_without_permission.md"
      ],
      "source_hash": "sha256:ghi...",
      "written_hash": "sha256:jkl...",
      "written_at": "2026-04-13T15:30:00",
      "last_sync": "2026-04-13T15:30:00",
      "status": "inbox"
    }
  }
}
```

**New status value:** `auto_promoted` — the plugin wrote the page directly to `shared/<type>/<slug>.md` without staging in `_inbox/`. Only applies to cluster types (`feedback`, `user`, `reference`). The existing `inbox`, `promoted`, `dismissed`, and `direct` (project) values are unchanged.

**New field:** `last_sync` — ISO timestamp of the most recent successful extraction run that wrote this entry. Used by `mnemo status` and by the single→multi upgrade detection. For v1 entries migrated forward, `last_sync` is initialized to `written_at`.

### 5.2 — Migration from schema v1

`schema_version=1` state files are migrated in-memory on load:

1. Parse as v1 format.
2. For each entry:
   - Copy `last_sync = written_at`.
   - Keep `status` as-is (no v1 entry had `auto_promoted` status).
3. Bump `schema_version = 2` in memory.
4. Next `atomic_write_state` persists v2 to disk.
5. No backup of the v1 file is written. The migration is idempotent and the next save is atomic — if it fails, the v1 file on disk is untouched.

`schema_version >= 3` on load still aborts with the v0.2 error message ("state file was written by a newer mnemo version"). Migration is strictly forward-compatible; there is no rollback.

### 5.3 — Extended decision table for cluster types

The v0.2 decision table §5.4 runs on every page. v0.3 extends it with a **target path selection step** that runs first:

```python
def _target_path_for_page(page: ExtractedPage, vault_root: Path) -> Path:
    if len(page.source_files) == 1:
        return vault_root / "shared" / page.type / f"{page.slug}.md"
    else:
        return vault_root / "shared" / "_inbox" / page.type / f"{page.slug}.md"
```

Once the target is chosen, the existing decision table runs, but with one important extension for the auto-promote branch:

| Status in state | Target path (from rule above) | On-disk hash == written_hash? | Classification | Action |
|---|---|---|---|---|
| absent | `shared/<type>/<slug>.md` | — | fresh_auto | write to `shared/<type>/<slug>.md`, record entry with `status=auto_promoted` |
| `auto_promoted` | `shared/<type>/<slug>.md` exists | yes | overwrite_safe | overwrite sacred file, update `written_hash`/`last_sync` |
| `auto_promoted` | `shared/<type>/<slug>.md` exists | **no** | sibling_bounced | write `shared/_inbox/<type>/<slug>.proposed.md`, leave sacred file intact, do NOT update `written_hash` of stored entry |
| `auto_promoted` | `shared/<type>/<slug>.md` absent | — | dismissed | user deleted the sacred file; transition entry to `dismissed`, skip |
| `inbox` | `shared/<type>/<slug>.md` | — | v0.2→v0.3 migration | legacy v0.2 entry for a page that is still single-source in v0.3; see §5.7 — classified as `fresh_auto`, status transitions `inbox → auto_promoted`, stale `_inbox/` file is left in place with an info log line |
| absent | `shared/_inbox/<type>/<slug>.md` | — | fresh_inbox | v0.2 behavior, unchanged |
| `inbox` / `promoted` / `dismissed` | `shared/_inbox/<type>/<slug>.md` | various | v0.2 rows §5.4 | unchanged |

**`sibling_bounced` is the critical new row.** It differs from v0.2's `sibling_proposed` only in the sibling path: v0.2 writes next to the target in `_inbox/`; v0.3 writes to `_inbox/` regardless of where the target lived. For multi-source pages (target already in `_inbox/`) the behavior is identical to v0.2. For auto-promoted pages (target in `shared/<type>/`), the sibling bounces back into `_inbox/` so the user finds it in their existing review surface.

### 5.4 — Decision table for project type (unchanged)

`shared/project/` behavior from v0.2 §5.4.1 is unchanged. Projects still bypass `_inbox/` and write directly to `shared/project/<agent>__<slug>.md`. The `direct` status is unchanged, independent from the new `auto_promoted` status. Sibling `.proposed.md` for project conflicts also stays in `shared/project/` (not bounced to `_inbox/`) — projects are agent-namespaced and the conflict path is already established.

**Rationale for not bouncing project siblings:** project pages are agent-scoped by filename (`<agent>__<slug>.md`), so the conflict is scoped to one agent and one file. Bouncing to `_inbox/project/<agent>__<slug>.proposed.md` would require a new subdirectory and a new review workflow for project conflicts, which is cost without benefit — the v0.2 pattern has been working and there is no dogfood evidence of discomfort with it.

### 5.5 — Post-process dedupe by slug (unchanged)

The v0.2 §5.5 dedupe pass still runs. It happens **before** `_target_path_for_page`, so a slug that was split across chunks has its `source_files` list already unioned by the time the target selection sees it. This means a slug with 1 source in chunk A and 1 source in chunk B (same conceptual rule, different files) correctly becomes a 2-source page after dedupe, correctly routes to `_inbox/`, and correctly does **not** auto-promote.

### 5.6 — The "single → multi upgrade" scenario

Run 1:
- Only one memory file exists for `use-yarn` (from `clubinho`).
- LLM produces single-source page.
- Target = `shared/feedback/use-yarn.md`, status = `auto_promoted`.

Run 2 (a week later):
- A new memory file appears (`central-inteligencia-frontend` also memorized a yarn rule).
- Scanner marks both files as dirty.
- LLM receives both and produces a 2-source clustered page with the same slug.
- `_target_path_for_page` says: `shared/_inbox/feedback/use-yarn.md`.

**Conflict:** the entry in state has `status=auto_promoted` and the file in `shared/feedback/use-yarn.md` already exists.

**Resolution (no regression, write to `_inbox/` as upgrade proposal):**

1. The auto-promote branch of the decision table does not match the target (target is `_inbox/`, not `shared/`), so it does not fire.
2. The `_inbox/` branch classifies the page as `fresh_inbox` from its perspective — no existing entry for `shared/_inbox/feedback/use-yarn.md` exists.
3. But before writing, the orchestrator checks: "is there an existing entry with the same slug and `status=auto_promoted`?" If yes:
   - Write the new clustered body to `shared/_inbox/feedback/use-yarn.proposed.md` (sibling, not fresh).
   - Do NOT create a new state entry with `status=inbox` — the existing `auto_promoted` entry is preserved intact.
   - Record the upgrade in the run summary as `upgrade_proposed`.
   - The user can diff `_inbox/feedback/use-yarn.proposed.md` against `shared/feedback/use-yarn.md`, decide if they accept the upgrade, and either manually merge into the sacred file or delete the proposal.

**Why preserve the existing entry intact:** transitioning from `auto_promoted` to `inbox` would mean the plugin decided to unpromote a page it had previously trusted. That's a regression in trust and would create confusion if the user had built workflows around the `shared/feedback/use-yarn.md` file. The no-regression rule is an invariant, encoded as a branch in `apply_pages`.

### 5.7 — Migration from v0.2 `inbox` entries that are still in `_inbox/`

A v0.2 user who upgrades to v0.3 without promoting any `_inbox/` pages will have state entries like `feedback/use-yarn` with `status=inbox`. On the first v0.3 run:

- The scanner sees the file in `_inbox/feedback/use-yarn.md`.
- The LLM re-emits the page (source is dirty because the scanner re-scans on schema migration — see §5.2 for the write-triggering behavior).

  **Correction:** the scanner does NOT force re-scan on migration. It only dirties files whose source hash differs from the stored `source_hash`. So if source is unchanged, nothing happens — the v0.2 entry with `status=inbox` is left alone until the next time the source memory file actually changes.

- When the source does change, the LLM re-emits. If the page is still single-source, `_target_path_for_page` says `shared/<type>/<slug>.md`. But the state entry says `status=inbox`, and there is no existing file at `shared/feedback/use-yarn.md`. This is classified as `fresh_auto` — the plugin writes to the sacred dir and transitions status `inbox → auto_promoted`.
- The old `shared/_inbox/feedback/use-yarn.md` file is **left in place** — the plugin does not delete files the user might have in review. Instead, the run summary reports "1 page upgraded from inbox to auto-promoted" and includes a hint line in the log: `ℹ cleanup: shared/_inbox/feedback/use-yarn.md is superseded by shared/feedback/use-yarn.md`.
- The user can delete the stale `_inbox/` file manually when they notice the log line. A future `mnemo doctor --cleanup` could automate this; deferred.

**Why not auto-delete the stale `_inbox/` file:** user data is sacred. The `_inbox/` directory is plugin-writable for writes, but deletes are user-only. Even when the plugin is confident the file is obsolete, the user may have pending edits in it or may be using it as a diff reference.

## 6. Components

All v0.3 components are **modifications** to existing files. No new sub-package is introduced. The module boundaries from v0.2 (`core/extract/`, `core/llm.py`, `hooks/session_end.py`, `cli.py`) carry over unchanged.

### 6.1 — Modified: `core/extract/scanner.py` (+ ~30 LOC)

Adds the schema v1 → v2 migration in `load_state`:

```python
def load_state(path: Path) -> ExtractionState:
    if not path.exists():
        return ExtractionState(schema_version=2, last_run=None, entries={})
    raw = json.loads(path.read_text(encoding="utf-8"))
    version = int(raw.get("schema_version", 1))
    if version == 1:
        raw = _migrate_v1_to_v2(raw)
    elif version > 2:
        raise ExtractionIOError(
            f"state file schema_version={version} was written by a newer mnemo version"
        )
    return _parse_state(raw)

def _migrate_v1_to_v2(raw: dict) -> dict:
    for key, entry in raw.get("entries", {}).items():
        entry.setdefault("last_sync", entry.get("written_at"))
        # status field is left alone — no v1 value maps to auto_promoted
    raw["schema_version"] = 2
    return raw
```

**`StateEntry` dataclass gains `last_sync: str` field.** Default value empty string. The `auto_promoted` status value is validated against a set of known values; unknown statuses log a warning and treat the entry as `dismissed` (conservative safe default).

### 6.2 — Modified: `core/extract/inbox.py` (+ ~90 LOC)

The densest change in v0.3. Three new functions plus extensions to `apply_pages`:

```python
def _target_path_for_page(page: ExtractedPage, vault_root: Path) -> Path:
    if len(page.source_files) == 1:
        return vault_root / "shared" / page.type / f"{page.slug}.md"
    return vault_root / "shared" / "_inbox" / page.type / f"{page.slug}.md"

def _is_auto_promoted_target(target: Path, vault_root: Path) -> bool:
    try:
        rel = target.relative_to(vault_root / "shared")
    except ValueError:
        return False
    return rel.parts[0] != "_inbox"

def _sibling_path(target: Path, vault_root: Path) -> Path:
    """Where does a .proposed.md sibling for this target live?

    Auto-promoted targets (in shared/<type>/) bounce their siblings back
    into shared/_inbox/<type>/.  _inbox/ targets keep siblings adjacent.
    """
    if _is_auto_promoted_target(target, vault_root):
        page_type = target.parent.name
        slug = target.stem
        return vault_root / "shared" / "_inbox" / page_type / f"{slug}.proposed.md"
    return target.parent / f"{target.stem}.proposed.md"
```

**`apply_pages` extension:** the classification loop now selects the target path first, then dispatches to either the auto-promote sub-path (new rows) or the inbox sub-path (v0.2 rows). The new rows are enumerated exhaustively in §5.3.

### 6.3 — Modified: `core/extract/__init__.py` (+ ~40 LOC)

The orchestrator gains two responsibilities:

1. **Background mode flag.** `run_extraction(cfg, *, dry_run=False, force=False, background=False)`. When `background=True`:
   - stdout/stderr are not printed (CLI layer redirects to `os.devnull`).
   - On completion, the `ExtractionSummary` is serialized to `~/mnemo/.mnemo/last-auto-run.json` via `atomic_write_state`-style write. On failure, the serialized summary still includes exit reason and exception type.
   - Error logging uses `where="extract.bg.*"` prefix so the circuit-breaker filter (§6.6) works.

2. **Post-apply delta into `ExtractionSummary`.** New fields:
   ```python
   @dataclass
   class ExtractionSummary:
       ... (v0.2 fields)
       auto_promoted: int = 0
       sibling_bounced: int = 0
       upgrade_proposed: int = 0

   ```
   The merge helper that folds `ApplyResult` into the summary handles the three new fields.

The top of `run_extraction` is unchanged — lock, scan, project promote. The cluster-type loop is unchanged structurally; only `apply_pages` behaves differently internally.

### 6.4 — Modified: `cli.py` (+ ~30 LOC)

New flag on the existing `extract` subcommand:

```
mnemo extract [--dry-run] [--force] [--background]
```

**`--background` behavior:**
- Redirects stdout and stderr to `os.devnull` for the duration of the run.
- Passes `background=True` to `run_extraction`.
- Never prints to stdout/stderr under any circumstance. Observable state lives in `last-auto-run.json` and `~/mnemo/.errors.log`.
- Exit codes follow v0.2 §6.7: 0 (success), 1 (partial failure), 2 (hard failure: lock held, CLI missing, state corrupt).
- **Writes to `last-auto-run.json` happen only from inside the `extract.lock` critical section** (inside `run_extraction`, after `locks.try_lock` succeeds). Failures that occur *before* the lock is acquired — most importantly, `ExtractionIOError` raised because the lock is already held by another run — are logged to `~/mnemo/.errors.log` with `where="extract.bg.lock"` but do NOT touch `last-auto-run.json`. This prevents contended runs from overwriting the success record of the winning run (see §7.6).
- Failures that occur *after* the lock is acquired (LLM subprocess errors, parse errors, disk full) write a `last-auto-run.json` record with `exit_code=1` and a populated `error` field, inside the lock, before releasing it.
- `KeyboardInterrupt` propagates normally and is not caught (typed-exceptions-only rule from v0.2).

**`--background` is not documented in `mnemo --help`** — it is an internal flag used by the `SessionEnd` hook. Listing it in the user-facing help would be misleading ("users shouldn't run this manually"). The flag is accepted silently when passed.

### 6.5 — Modified: `hooks/session_end.py` (+ ~60 LOC)

The v0.2 `_maybe_emit_hint` is replaced by a new `_maybe_schedule_extraction` that handles both the opt-in auto-brain path and the legacy hint path:

```python
def _maybe_schedule_extraction(cfg, vault_root, agent_name):
    try:
        auto_enabled = bool(cfg.get("extraction", {}).get("auto", {}).get("enabled", False))
        if not auto_enabled:
            _maybe_emit_hint_legacy(cfg, vault_root, agent_name)
            return

        state_path = vault_root / ".mnemo" / "extraction-state.json"
        if not _debounce_passes(state_path, cfg):
            return

        lock_path = vault_root / ".mnemo" / "extract.lock"
        if _lock_held(lock_path):
            return

        _spawn_detached_extraction()
    except Exception as exc:
        errors.log_error(vault_root, "session_end.schedule", exc)
        # Never re-raise. Hook cannot fail the session.
```

**`_spawn_detached_extraction` is the cross-platform core:**

```python
def _spawn_detached_extraction() -> None:
    kwargs = dict(
        stdin=subprocess.DEVNULL,
        stdout=subprocess.DEVNULL,
        stderr=subprocess.DEVNULL,
        close_fds=True,
    )
    if sys.platform == "win32":
        CREATE_NEW_PROCESS_GROUP = 0x00000200
        DETACHED_PROCESS = 0x00000008
        kwargs["creationflags"] = DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP
    else:
        kwargs["start_new_session"] = True
    argv = [sys.executable, "-m", "mnemo", "extract", "--background"]
    subprocess.Popen(argv, **kwargs)
```

**`_debounce_passes` is a pure function** that reads the state file and checks both count and time conditions. Extracted for unit testability.

**`_lock_held` is also a pure function** — checks if `extract.lock` exists as a directory and is less than 5 minutes old (matching the `locks.try_lock` stale-detection logic). If stale, returns False (the new run will reclaim it).

**`_maybe_emit_hint_legacy` is the v0.2 `_maybe_emit_hint` verbatim**, kept for the case where the user has `auto.enabled=false`. The logic is unchanged; only the entry point renames.

**Error handling:** the entire function is wrapped in `try/except Exception` (not bare `except:` — `KeyboardInterrupt` and `SystemExit` still propagate). Any exception is logged via `errors.log_error` with `where="session_end.schedule"` and silenced. The hook returns 0 regardless.

### 7.4 — Single → multi upgrade

Day 1: Only one memory for `use-yarn`, auto-promoted to `shared/feedback/use-yarn.md`.

Day 10: A second agent memorized a yarn-related rule. Both memories are dirty on the next auto-run.

```
LLM receives both files, produces 2-source clustered page for slug "use-yarn".
apply_pages:
  target = shared/_inbox/feedback/use-yarn.md (multi-source → _inbox branch)
  pre-check: existing state entry "feedback/use-yarn" with status=auto_promoted
             → upgrade detected
  classification: upgrade_proposed
  write shared/_inbox/feedback/use-yarn.proposed.md with the new clustered body
  do NOT create a new state entry
  do NOT change status on the existing entry
  shared/feedback/use-yarn.md UNTOUCHED
  summary.upgrade_proposed += 1

last-auto-run.json records:
  "upgrade_proposed": [{"slug": "feedback/use-yarn", "sibling": "shared/_inbox/feedback/use-yarn.proposed.md"}]
```

The user decides whether the clustered upgrade is worth merging into their sacred file. If yes, they overwrite `shared/feedback/use-yarn.md` by hand and delete the `.proposed.md`. Next run's content-addressing detects the new content (`on-disk hash != written_hash`) and the existing `auto_promoted` status is preserved because the apply_pages logic continues to see the same slug and single-source target would fire `sibling_bounced` — wait, this case needs care.

**Subtle case worth expanding:** after the user merges the upgrade by hand into `shared/feedback/use-yarn.md`, the next auto-run sees:
- `on-disk hash` of `shared/feedback/use-yarn.md` != stored `written_hash` (user edited).
- But the source is also dirty (both agents' memories are still present).
- LLM produces a 2-source page again; target classification says `_inbox/`.
- `apply_pages` enters the multi-source branch, finds the existing `auto_promoted` entry for this slug, classifies as `upgrade_proposed` AGAIN.
- Another sibling is written to `_inbox/`, but the user just merged that content. They see the sibling and either delete it or realize it's redundant.

**This is acceptable but annoying, and explicitly deferred to v0.4.** The repeated sibling proposals are harmless: the sacred file is never touched, the state is never corrupted, the user's merged content persists. The only cost is noise in the `_inbox/` directory each run until the user re-edits the sacred file in a way that moves its hash forward (e.g., accepts a future upgrade, or fixes a typo).

v0.3 does NOT implement a "merged upgrade" detection. v0.4 will consider a `last_upgrade_hash` field on state entries that records the sibling's hash; if a subsequent run sees the sacred file's on-disk hash match `last_upgrade_hash`, it interprets this as "user merged my proposal" and updates `written_hash` + `source_files` accordingly. Deferred because:

1. The corner case requires the specific sequence (multi-source upgrade → user merges by hand → LLM produces the same multi-source cluster on the next run), which is rare in practice.
2. Adding the field and the detection logic expands the decision table and adds a state-schema field that doesn't benefit the common path.
3. The v0.3 scope is already focused; v0.4 has room to polish.

If dogfood reveals this is actually noisy, file a v0.4 issue with the observed frequency.

### 7.6 — Concurrent SessionEnds

Two Claude Code sessions (two terminals) end within milliseconds of each other. Both hooks run `_maybe_schedule_extraction`. Both pass the debounce check (both see same `last_run`). Both call `_lock_held` (both see no lock). Both spawn `Popen`.

- Subprocess A acquires `extract.lock` first.
- Subprocess B calls `locks.try_lock`, sees A's lock directory, returns False.
- Subprocess B's `run_extraction` raises `ExtractionIOError("another extraction is in progress")`.
- Subprocess B's `--background` CLI layer catches it, writes `last-auto-run.json` with `exit_code: 2` and `error: {"type": "ExtractionIOError", "message": "another extraction is in progress"}`.
- Subprocess B exits 2.

**Problem:** B's `last-auto-run.json` write OVERWRITES A's successful write once A finishes. The user's `mnemo status` would then see B's failure record instead of A's success.

**Mitigation:** only write `last-auto-run.json` from inside the lock. B never enters the lock, so B never writes the file. A writes it inside its critical section, and B's failure is logged to `.errors.log` under `where="extract.bg.lock"` but does not touch `last-auto-run.json`.

**Revised spec of `--background` error handling:** the CLI layer catches `ExtractionIOError` caused by lock contention specifically, logs it via `errors.log_error`, and exits 2 without writing `last-auto-run.json`. Any other error (subprocess failure, JSON parse error, disk full) is still written to `last-auto-run.json`. This is the simplest correct behavior and preserves the A-success view in `mnemo status`.

## 8. Error handling and robustness

### 8.1 — Governing rules (unchanged)

- **Hooks never crash the session.** v0.1 and v0.2 rule. v0.3's `_maybe_schedule_extraction` is wrapped in `try/except Exception` with silent fallback.
- **Manual commands may fail loudly.** v0.2 rule. `mnemo extract` without `--background` still prints errors to stderr.
- **Typed exceptions only in `core/extract/`.** v0.2 rule. `KeyboardInterrupt` propagates.

### 8.2 — New rule for v0.3: background commands fail silently

`mnemo extract --background` must produce **zero output** on stdout and stderr under all conditions. All feedback goes through:

1. `~/mnemo/.mnemo/last-auto-run.json` — the success/failure summary.
2. `~/mnemo/.errors.log` — detailed error records for `mnemo doctor`.
3. `extract.lock` — absence after the run is the "I finished" signal.

The rationale is that a detached subprocess has no UI. If it printed to stderr, nothing would see it (the parent has already returned). The only observers are future `mnemo status`, `mnemo doctor`, and user introspection of the files.

### 8.3 — Failure domain table (delta from v0.2)

| Failure | v0.3 action |
|---|---|
| `Popen` raises `OSError` (permission, too many fds) | Caught in `_maybe_schedule_extraction`'s outer try/except, logged via `errors.log_error(where="session_end.schedule.popen")`, hook returns 0. The background process never started. |
| `Popen` succeeds but child `python -m mnemo` cannot be launched (no interpreter, corrupt install) | Parent has already returned. Child exits with its own error. Nothing is logged because the child never got to the `errors.log_error` call site. **Accepted failure mode.** `mnemo doctor` will eventually notice "no auto-runs in N days" and alert. |
| Schedule passes debounce but `extract.lock` contended (another run started 50ms earlier) | `_lock_held` returns True, schedule skips, no spawn. On the next SessionEnd the same check runs and either succeeds or skips again. |
| Background `run_extraction` crashes mid-chunk | v0.2 partial-success semantics apply. `last-auto-run.json` is written with exit code 1. `extract.lock` is released on context-manager exit. |
| `last-auto-run.json` write fails | Logged via `errors.log_error(where="extract.bg.summary")`. `extract.lock` still released. The file on disk may be corrupt or absent — `mnemo status` treats corrupt/missing as "no prior run". |
| v1 → v2 state migration fails (invalid v1 JSON) | Handled by existing v0.2 code: rename to `.bak.{timestamp}`, treat as empty state, log warning. Run proceeds fresh. |
| User flipped `auto.enabled=true` without v0.2.1 | No runtime failure. `mnemo doctor` emits the advisory warning (§6.8). Runs proceed. Bad clustering pollutes sacred dir — this is the risk v0.2.1 is supposed to prevent, and the spec is explicit that landing v0.2.1 first is the user's responsibility. |

### 8.4 — Concurrency and the `last-auto-run.json` race

See §7.6. The rule is: **`last-auto-run.json` is only written from inside the lock.** Contended schedules do not touch the file. This eliminates the race.

### 8.5 — Circuit breaker interaction

- `session_end.schedule` errors are **not** counted by the circuit breaker (new exception in `core/errors.py::should_run`).
- `extract.bg.*` errors are **not** counted (v0.2 filter already in place for `extract.*` prefix).
- Hook body errors outside extraction (mirror, log_writer) still count normally.

### 8.6 — Explicitly not handled

- **User flips `auto.enabled` mid-session and the first SessionEnd of that session fires.** The config read at hook time picks up the new value. Works correctly.
- **User renames a file in `shared/<type>/` to take it out of auto-promote tracking.** The state entry still references the old filename. Next run cannot find the file, treats as `dismissed`. The renamed file is effectively a user-promoted copy (no `auto-promoted` tag in frontmatter means the plugin never touches it). **This is the correct escape hatch** — renaming is how the user "adopts" a page out of auto-promote territory.
- **User writes a page to `shared/feedback/<slug>.md` by hand that later matches an LLM-generated slug.** Plugin reads existing file's frontmatter, sees no `auto-promoted` tag, treats the slot as user-owned, stages the plugin version in `_inbox/` instead. No collision in `shared/`. (This row needs a test case; see §9.3.)

## 9. Testing strategy

### 9.1 — Inherited from v0.2

All v0.2 tests continue to pass unchanged (regression bar). The layering (`tests/unit/`, `tests/integration/`, `tests/e2e/`) and the "single seam at `core/llm.py::call`" principle still apply.

### 9.2 — New unit test modules

**`tests/unit/test_state_schema_v2.py`**
- Load v1 file → in-memory state has `schema_version=2` and `last_sync` populated from `written_at`.
- Load v2 file → unchanged round-trip.
- Load v3 file → `ExtractionIOError`.
- Load corrupt JSON → rename to `.bak`, empty state returned.
- Round-trip: load v1, add entry, save, reload → saved file is v2, new entry has `last_sync`.

**`tests/unit/test_extract_inbox_v0_3.py`**
- `_target_path_for_page`: 1-source page → sacred path; 2-source page → inbox path; 3+ source → inbox path.
- `_is_auto_promoted_target`: sacred path → True; inbox path → False; project path → False (project has its own flow).
- `_sibling_path`: sacred target → sibling in `_inbox/`; inbox target → sibling adjacent.
- `apply_pages` fresh_auto: 1-source fresh page → file written to `shared/feedback/`, state entry `status=auto_promoted`, frontmatter has `tags: [auto-promoted]`.
- `apply_pages` overwrite_safe on auto_promoted: source changed, on-disk hash matches → overwrite sacred file, update `written_hash` + `last_sync`.
- `apply_pages` sibling_bounced: source changed, on-disk hash mismatches → sibling written to `_inbox/feedback/<slug>.proposed.md`, sacred file untouched, state unchanged.
- `apply_pages` upgrade_proposed: existing entry `status=auto_promoted`, new page has 2 sources → sibling written to `_inbox/`, existing entry preserved, new state entry NOT created.
- `apply_pages` dismissed of auto_promoted: state says `auto_promoted`, sacred file deleted by user → transition to `dismissed`, skip.
- Frontmatter emission: auto-promoted pages have `tags: [auto-promoted]` and `last_sync`; inbox pages have `tags: [needs-review]` and no `last_sync` change on write.
- User-owned file at `shared/feedback/<slug>.md` with no `auto-promoted` tag, plugin wants to write same slug → falls through to `_inbox/` (treats sacred slot as taken).

**`tests/unit/test_session_end_schedule.py`**
- `_debounce_passes`: all 4 combinations of (count pass/fail) × (time pass/fail). Only the double-pass fires.
- `_lock_held`: directory absent → False; directory fresh → True; directory stale (>5min old) → False.
- `_spawn_detached_extraction`: monkeypatch `subprocess.Popen`, assert argv, assert POSIX flags (`start_new_session=True`), assert Windows flags (`creationflags=DETACHED_PROCESS|CREATE_NEW_PROCESS_GROUP`).
- `_maybe_schedule_extraction`:
  - `auto.enabled=false` → falls through to legacy hint, no Popen.
  - `auto.enabled=true`, debounce fails → no Popen, no error.
  - `auto.enabled=true`, debounce passes, lock held → no Popen, no error.
  - `auto.enabled=true`, debounce passes, lock free → Popen called.
  - Popen raises `OSError` → caught, logged with `where="session_end.schedule.popen"`, hook returns 0.
  - Arbitrary exception inside the body → caught by outer `try/except Exception`, logged, returns 0.

**`tests/unit/test_cli_background.py`**
- `mnemo extract --background` redirects stdout/stderr to DEVNULL (monkey-patch and assert).
- On success: writes `last-auto-run.json` with exit_code=0 and full summary.
- On `LLMSubprocessError`: writes with exit_code=1, `error` field populated.
- On `ExtractionIOError` from lock contention: does NOT write `last-auto-run.json`; logs to `extract.bg.lock`; exits 2.
- On unexpected `Exception`: writes with exit_code=1, `error` field populated.
- `KeyboardInterrupt` propagates (no catch).

**`tests/unit/test_doctor_status_v0_3.py`**
- `mnemo status` with no `last-auto-run.json` and auto disabled → "disabled" line.
- `mnemo status` with `last-auto-run.json` success → "last run: Xm ago — N written".
- `mnemo status` with `extract.lock` held → "running now: PID X, started Ys ago".
- `mnemo doctor` with failed `last-auto-run.json` → warning line.
- `mnemo doctor` with stale lock >10min → warning line.
- `mnemo doctor` with `auto.enabled=true` and no `last-auto-run.json` older than 7 days → "auto-brain has not run successfully" advisory.

### 9.3 — New integration tests

**`tests/integration/test_auto_brain_pipeline.py`**
- Fixture: tmp vault with 5 memory files (3 feedback + 2 user), all types.
- Mock `llm.call` with a response that produces 1 single-source feedback + 1 multi-source feedback + 1 single-source user.
- Run `run_extraction(background=True)` end-to-end.
- Assert: `shared/feedback/<single>.md` exists with `auto-promoted` tag; `shared/_inbox/feedback/<multi>.md` exists with `needs-review` tag; `shared/user/<single>.md` exists with `auto-promoted` tag.
- Assert `last-auto-run.json` exists with the correct summary counts.
- Second run without source changes → zero writes, `last-auto-run.json` updated with zero counts.
- Third run: user edits the sacred single-source page (modify file on disk), source memory changes → sibling_bounced, sacred file untouched, `_inbox/` has `.proposed.md`.
- Fourth run: upgrade_proposed path — second agent memory added, now 2-source cluster on same slug, existing auto-promoted entry preserved, sibling written to `_inbox/`.

**`tests/integration/test_session_end_to_extraction.py`**
- Full SessionEnd hook body with mocked config (`auto.enabled=true`).
- Monkey-patch `subprocess.Popen` to record the argv and flags.
- First run (no prior state) → passes debounce (last_run is None → elapsed=inf), spawns.
- Second run within 60min → no spawn.
- Third run after 61min → spawns again.
- `auto.enabled=false` → no spawn, legacy hint line emitted.
- Circuit breaker tripped → no schedule body runs.

### 9.4 — Smoke tests (opt-in, `MNEMO_E2E=1`)

Extend `tests/e2e/test_real_llm.py` with one new case:

- Tmp vault with 1 single-source feedback file.
- `auto.enabled=true`, `minNewMemories=1`, `minIntervalMinutes=0`.
- Fire `run_extraction(background=True)` directly.
- Assert: `shared/feedback/<slug>.md` exists with `auto-promoted` tag, `last-auto-run.json` has success, `extract.lock` released.

No test of the detached `Popen` path — that would require spawning real subprocesses in CI, which is flaky. Unit tests with monkey-patched `Popen` cover the argv/flags contract.

### 9.5 — Coverage targets

- `hooks/session_end.py::_maybe_schedule_extraction` — 100% branch coverage.
- `core/extract/inbox.py` — 100% of the extended decision table (every new row).
- `core/extract/scanner.py` migration — 100% of the migration function.
- `cli.py --background` handler — 100% of the four failure modes.

### 9.6 — What we explicitly do not test

- **Detached subprocess survival across parent exit.** Requires real OS fork semantics; covered by manual dogfood, not CI.
- **Real time-based debounce** (waiting 60 minutes in a test). Tests use monkey-patched clocks.
- **Subprocess argument quoting on Windows.** Uses stdlib `subprocess.list2cmdline` semantics; trust Python's implementation.
- **The "user merged the upgrade proposal manually" detection** — not implemented in v0.3, deferred to v0.4 (see §7.4). The v0.3 test suite verifies that the sibling is re-emitted each run in this scenario, not that it is suppressed.

## 10. Open questions (for v0.4+)

v0.3 deliberately leaves these for later:

1. **Default flip of `auto.enabled=true`.** After 2-4 weeks of real use, audit the failure-to-success ratio and the user-conflict count. If auto-brain is trusted, flip default in v0.4.
2. **Log-based extraction.** v0.2 non-goal that is still a non-goal in v0.3. v0.4 may revisit.
3. **Interactive merge tool.** `mnemo review` that walks the user through `_inbox/` pending items and offers keep/discard/merge. Becomes valuable when `_inbox/` regularly has 5+ items waiting.
4. **Extraction quality telemetry.** `mnemo doctor --extraction` showing acceptance ratio over 30 days. Depends on v0.3 having run long enough to have data.
5. **Semantic deduplication of `_inbox/`.** If the user leaves a multi-source cluster in `_inbox/` for weeks and the plugin re-emits the same cluster on every run, can we detect "already seen, no user action" and suppress the re-emission? Subtle; may regress the content-addressing guarantee.
6. **Cross-vault sync.** Still deferred to "user uses git/Syncthing".
7. **`mnemo review` CLI flow.** Would replace manual Obsidian navigation of `_inbox/` with a guided console flow. Low priority; Obsidian works well enough.

## 11. Design decisions log

Decisions made during the v0.3 brainstorm. Numbering continues from v0.2's log (#30 was the last v0.2 entry).

| # | Decision | Rationale |
|---|---|---|
| 31 | v0.3 scope = auto-brain mode only (Eixos A + B); v0.2.1 (prompt fixes) is a separate release and hard prerequisite | Split keeps each release testing one change; v0.2.1 can be dogfooded independently before auto-promote is enabled on top of it |
| 32 | Auto-promote rule: single-source → direct to `shared/<type>/`; multi-source → stage in `shared/_inbox/<type>/` | Aligns review surface with decision surface — plugin is accountable when it makes merge calls, not when it moves single files |
| 33 | Conflict siblings for auto-promoted pages bounce back to `_inbox/` | `_inbox/` becomes the unified review surface; sacred `shared/<type>/` stays uncontaminated by plugin artifacts |
| 34 | Single → multi upgrade: no regression, write sibling to `_inbox/` | Once a slug is auto-promoted, it stays auto-promoted; upgrades are proposals, not demotions |
| 35 | `auto.enabled=false` default on v0.3 ship | Opt-in is safer for the first release that makes LLM calls without user initiation; user builds trust first |
| 36 | Background trigger = fire-and-forget `subprocess.Popen` with detach flags | Stdlib-only, cross-platform, no daemon; reuses existing `run_extraction` verbatim |
| 37 | Debounce = `count >= minNewMemories AND elapsed >= minIntervalMinutes` (AND gate) | Single-knob approaches either thrash (count-only) or waste LLM calls (time-only); AND gate prevents both |
| 38 | `minIntervalMinutes` default = 60 | Matches natural "deep-work cycle" — one extraction per work block, runs while user is away from keyboard |
| 39 | Lock primitive unchanged — `locks.try_lock` already serializes all the v0.3 cases | Atomic mkdir + self-heal covers every concurrency scenario without new primitives |
| 40 | `last_run` advances on success only (inherited from v0.2) | Transient failures retry on the next debounce tick; persistent failures surface via `mnemo doctor` |
| 41 | `last-auto-run.json` written only from inside the lock | Prevents contended schedules from overwriting the success record of the winning run |
| 42 | New status value: `auto_promoted` (not `auto`, not `direct_promoted`) | Aligns with frontmatter `auto-promoted` tag; expresses intent ("approval was bypassed") not mechanism |
| 43 | `auto-promoted` is emitted as an Obsidian tag inside `tags:`, not as a scalar `auto_promoted: true` | Maximizes Obsidian usability — `tag:#auto-promoted` query returns every plugin-written page; graph view shows them as a cluster |
| 44 | `last_sync` added to state entries (not just `written_at`) | Separates "when the plugin last touched this page" from "when this page was first written to disk"; future telemetry will use `last_sync` for drift detection |
| 45 | State schema bumped to version 2 with in-memory migration from v1 | Forward-only; no backup of v1 file; migration is idempotent and tested |
| 46 | `--background` CLI flag is hidden from `mnemo --help` | Internal contract between `session_end.py` and `cli.py`; users should never invoke it manually |
| 47 | `mnemo status` and `mnemo doctor` both read `last-auto-run.json` but with different depth | Status = overview panel, Doctor = active diagnostic alerts; same source of truth, different presentations |
| 48 | No `mnemo doctor` runtime check for v0.2.1 prerequisites | No reliable string-free way to detect whether installed prompts include the #5 fix; enforcement is a release-process gate (§12 milestone 8), not a runtime check |

## 12. Milestones and deliverables

v0.3 is a single release. Milestones in strict dependency order:

1. **State schema v2 + migration (`core/extract/scanner.py`).** Fundação — everything downstream reads the state through this path. Tests: `test_state_schema_v2.py`.
2. **`inbox.py` decision table extension.** The core business rule. Tests: every new row of §5.3 plus the user-owned-slot edge case (§8.6). Must pass before any background wiring lands.
3. **`cli.py --background` flag.** Hide from help, redirect IO, write `last-auto-run.json`. Tests: `test_cli_background.py`.
4. **`hooks/session_end.py::_maybe_schedule_extraction`.** Debounce + spawn. Monkey-patched Popen tests. Integration test with mocked LLM.
5. **`mnemo status` and `mnemo doctor` v0.3 sections.** Reads the files the previous milestones wrote. Tests: `test_doctor_status_v0_3.py`.
6. **`core/errors.py` filter update + `core/config.py` schema.** One-liners, bundled with integration pass.
7. **Integration test end-to-end:** `test_auto_brain_pipeline.py` + `test_session_end_to_extraction.py`.
8. **Dogfood preparation.** Author validates v0.2.1 is installed and clustering is fixed via one manual `mnemo extract` on the real vault. Compare output to the current 7-page baseline.
9. **Dogfood week.** Flip `auto.enabled=true` on the author's vault. Run normally for 5-7 days. Track: conflicts, upgrade proposals, false-positive auto-promotes, SessionEnd overhead.
10. **Bugfixes from dogfood.** Any regression is blocking. Any new edge case is a GitHub issue.
11. **Update `README.md` and `docs/getting-started.md`** with auto-brain usage, the opt-in flag, and the v0.2.1 prerequisite.
12. **Tag v0.3.0, ship.**

Each milestone produces a green test run before the next begins. The implementation plan (produced by `writing-plans`) will define per-milestone acceptance criteria.

## 13. File formats

### 13.1 — Auto-promoted page format

```markdown
---
name: Use yarn
description: Package manager rule for the clubinho project
type: feedback
extracted_at: 2026-04-13T15:30:00
extraction_run: 2026-04-13T15:30:00-abc123
last_sync: 2026-04-13T15:30:00-abc123
sources:
  - bots/clubinho/memory/feedback_use_yarn.md
tags:
  - auto-promoted
---

Always use yarn for package management in this project.

**Why:** Mixing yarn and npm causes lockfile conflicts that broke the CI twice.

**How to apply:** Run `yarn install`, `yarn add`, `yarn run`. Never run `npm install` or `npx` in the project root.
```

**Key differences from v0.2 inbox pages:** `tags: [auto-promoted]` (not `needs-review`), `last_sync` field present.
