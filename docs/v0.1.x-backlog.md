# v0.1.x backlog

Items discovered during v0.1 dogfood that should be triaged for the next
patch release. When a fresh Claude session starts work on v0.1.x or v0.2,
**scan this file first** to know what's already known.

Each item is tagged with where it came from:

- `[dogfood]` — found by the author using mnemo on their own machine
- `[spec]` — listed in the v0.1 design spec but never implemented
- `[review]` — surfaced during code review but not fixed in v0.1.0

---

## Already fixed in v0.1.0 (post-launch hotfixes)

- ✅ **Agent name resolution** (PR #2). `_agent_from_project_dir` heuristic
  produced wrong names for repos in nested layouts (e.g.,
  `~/github/refactor/sg-imports` → `refactor-sg-imports` instead of
  `sg-imports`). Fixed by reverse-decoding the encoded path via filesystem
  walk and delegating to `agent.resolve_agent`.
- ✅ **Hook command marker corruption** (PR #3). `MNEMO_TAG = "mnemo:"` was
  prepended to the command field, breaking shell dispatch. Hooks never
  ran. Fixed by changing the marker to `"mnemo.hooks."` (a substring
  naturally present in every command we emit).

Both bugs were discovered within the first 30 minutes of real dogfood,
which validates the spec § 11.2 release-gate decision to require
self-dogfood before tagging v0.1.0.

## Open issues

### `[dogfood]` 1. PostToolUse log noise

30 edits to the same file in one session produce 30 log lines. Spec § 12.3
originally scheduled debouncing for v0.3, but the noise is visible
immediately on real usage. **Consider promoting to v0.1.1.**

**Suggested fix:** track the most recent `(agent, file_path)` pair in the
session cache; if the same pair appears within N seconds (e.g., 30s),
suppress the log line. Reset the dedup window when a different file is
edited.

The dedup state should live in the session cache (already per-session),
not in a module global, so concurrent sessions don't interfere.

---

### `[dogfood]` 2. Cosmetic: `prompt_input_exit` reason string

SessionEnd log lines come out as `🔴 session ended (prompt_input_exit)`.
The reason name is Claude Code's internal enum and looks ugly.

**Suggested fix:** small lookup table in `hooks/session_end.py`:

```python
_REASON_LABELS = {
    "prompt_input_exit": "exit",
    "user_exit": "exit",
    # add other internal reasons as we discover them
}
reason = _REASON_LABELS.get(raw_reason, raw_reason)
```

Don't enumerate every possible Claude Code reason — just the common ones
that come up. Unknown reasons fall through to the raw string.

### `[review]` 3. Naming heuristic edge cases

The `_decode_path` filesystem walk in `core/mirror.py` works for typical
layouts but might have false positives or false negatives for:

- **Symlinked project directories.** `Path.exists()` follows symlinks; the
  decode might end up at a different real path than the user expects.
- **Mount points** (e.g., `~/Volumes/external/repo` on macOS).
- **Repos under directories with literal `-` in the name** that ALSO
  conflict with sibling-name disambiguation. The greedy depth-first
  search picks the first deepest match, which could be wrong if two
  valid decodings exist.
- **Windows native paths.** We never validated decoding on a `C:\...`
  style path. The Windows job is `continue-on-error` so this isn't
  blocking, but expect issues if a Windows user dogfoods.

No reproducer exists yet. Add a test the moment a real issue surfaces.

### `[dogfood]` 4. Documentation gap: install path

`docs/getting-started.md` shows `/plugin install mnemo@claude-plugins-official`
as the primary install path, but the plugin **is not actually published**
in that marketplace yet. Two options:

1. **Publish to the marketplace** (this is the M8 launch task in the spec).
   Then the docs are accurate.
2. **Update the docs** to lead with the manual `git clone + pip install -e .`
   path until publication happens.

Pick (1) as part of the v0.1.0 launch checklist.

---

### `[dogfood]` 5. Stale `bots/` directories from old runs

When the agent name resolution fix landed (PR #2), it left orphan `bots/<wrong-name>/`
directories from prior runs that used the broken heuristic. The user had to
manually `rm -rf` them. We could add a `mnemo doctor --suggest-cleanup` mode
that detects bot dirs which no longer match any current Claude Code project
and suggests removal — but this is low-priority polish.

**Decision:** not worth automating in v0.1.x. Document it in
troubleshooting if a user reports confusion.

## Things explicitly NOT in scope for v0.1.x

These came up in conversation but were explicitly deferred or rejected:

- **Migrating the legacy `~/obsidian-memory/` vault.** That's the author's
  personal data from the pre-mnemo prototype. It's preserved as a frozen
  historical archive. Not mnemo's responsibility.
- **Writing a `--bare` mode for v0.2 LLM extraction.** The `--bare` flag of
  `claude` is incompatible with subscription auth, which would defeat the
  whole point of the hybrid approach. See spec § 12.1 and decision #16.
- **Auto-promotion from Tier 1 to Tier 2.** That's the v0.2 LLM extraction
  feature. Doing it manually is a v0.1 feature on purpose (spec § 4.4
  rule of trust: "plugin never auto-promotes across tier boundaries in
  v0.1").
